## Phase 1：Schema 与通用流程

> 参考：[实施计划 → Phase 1](./14-unified-source-metadata-plan.md#phase-1schema-与通用流程)

### 1.1 ColumnInfo 扩展

- `ColumnInfo` 增加可选 `description` 字段
— 参考 [元数据模型 → 列级元数据](./14-unified-source-metadata-plan.md#列级元数据) 与 [Workspace 持久化](./14-unified-source-metadata-plan.md#workspace-持久化)
- 旧 metadata 反序列化兼容：没有 `description` 时正常加载
- 序列化时只有存在 `description` 才写入（不写 `null`）

### 1.2 list_tables() 轻量化改造

- PostgreSQL `list_tables()` 去掉 `SELECT * LIMIT 10` 和 `SELECT COUNT(*)`
- MySQL `list_tables()` 同上
- 其他已有 Loader（MSSQL / Kusto / Athena 等）视情况同步轻量化
- 改造后 `list_tables()` 只查 `information_schema`，返回表名、列名、列类型
— 参考 `[list_tables()` 轻量化改造](./14-unified-source-metadata-plan.md#list_tablestable_filternone-轻量化改造)

### 1.3 get_column_types() 默认行为更新

- 默认实现保留 `get_metadata(path)` 返回的表级 `description`
— 参考 `[get_column_types(source_table)](./14-unified-source-metadata-plan.md#get_column_typessource_table)`

### 1.4 ingest_to_workspace() 元数据合并

- Import 流程中 best-effort 调用 `get_column_types(source_table)`
- 合并表描述 → `TableMetadata.description`
- 合并列描述 → `ColumnInfo.description`
- metadata 增强失败时不阻断导入
— 参考 [数据流 → Import](./14-unified-source-metadata-plan.md#import)

### 1.5 /api/tables/list-tables 扩展

- 返回体增加 `table.description`
- 返回体列信息增加 `column.description`
— 参考 [API 变化 → /api/tables/list-tables](./14-unified-source-metadata-plan.md#apitableslist-tables)

### 1.6 Phase 1 测试

- `ColumnInfo` 反序列化旧 metadata 无 description 成功
- `ColumnInfo` 仅有 description 时才序列化
- `ingest_to_workspace()` metadata 增强抛错仍成功导入
- `get_column_types()` 默认实现保留 `description`
- `/api/tables/list-tables` 有描述时返回表/列描述
— 参考 [测试策略 → 单元测试](./14-unified-source-metadata-plan.md#测试策略)

## Phase 2：高价值 Loader

> 参考：[实施计划 → Phase 2](./14-unified-source-metadata-plan.md#phase-2高价值-loader) 与 [各数据源映射](./14-unified-source-metadata-plan.md#各数据源映射)

### 2.1 PostgreSQL

- `get_metadata(path)` / `get_column_types()` 查 `obj_description` / `col_description`
- 只查选中表，不扫描全 schema
- 轻量化后的 `list_tables()` 同步返回表/列描述

### 2.2 MySQL

- 查 `information_schema.TABLES.TABLE_COMMENT` 和 `COLUMNS.COLUMN_COMMENT`
- 空字符串归一化为 `None`

### 2.3 BigQuery

- 使用 SDK 已有的 `table.description` 和 `field.description`

### 2.4 Superset

- Dataset 级和列级 `verbose_name` / `description`
- 保留 `is_dttm` 和类型归一化
- 多层描述拍平为一个 markdown 章节 `description`

### 2.5 Phase 2 测试

- 测试数据库包含表/列 comment 时拉取正确
- metadata 查询失败不影响 preview/import
- 空 comment 归一化为 `None`
— 参考 [测试策略 → Loader 测试](./14-unified-source-metadata-plan.md#测试策略)

## Phase 3：其他 Loader

> 参考：[实施计划 → Phase 3](./14-unified-source-metadata-plan.md#phase-3其他-loader)

- ✅ MSSQL — 表/列级 `MS_Description` 扩展属性，优先使用参数化查询
- ✅ Kusto — 表 DocString / 列 DocString
- ✅ Athena/Glue — Glue table description / column comments（可能需 Glue API）
- ⏭️ MongoDB / CosmosDB — 跳过；无原生 schema description，保持现状
- ⏭️ S3 / Azure Blob / Local Folder — 跳过；文件系统无表/列描述，保持现状

## Phase 4：UX 展示与 Catalog 虚拟化

> 参考：[实施计划 → Phase 4](./14-unified-source-metadata-plan.md#phase-4ux-展示与-catalog-虚拟化) 与 [前端行为](./14-unified-source-metadata-plan.md#前端行为)

### 4.1 Catalog 树 UI

- 存在表/数据集描述时，tooltip 或详情面板展示
- metadata 加载中不阻塞 preview/import
- 不强制为所有可见节点拉取详细 metadata
- `source_metadata_status` 状态指示器
— 参考 [表级元数据 → source_metadata_status](./14-unified-source-metadata-plan.md#表级元数据)

### 4.2 Preview Panel UI

- 表描述展示在 preview 标题附近
- 列描述 tooltip 或紧凑描述列
- 筛选和 row limit 行为不变
— 参考 [API 变化 → /api/connectors/preview-data](./14-unified-source-metadata-plan.md#apiconnectorspreview-data)

### 4.3 已导入 Workspace 表 UI

- 系统描述（`TableMetadata.description`）只读展示
- 用户描述（`attachedMetadata`）可编辑
- 同时展示两个描述，互不覆盖
— 参考 [前端行为 → 已导入 workspace 表](./14-unified-source-metadata-plan.md#前端行为)（两个独立描述字段图示）

### 4.4 Catalog 树虚拟化

- 替换 MUI `SimpleTreeView` + `TreeItem` 为虚拟滚动方案
- 推荐方案：`react-arborist`；备选：`@tanstack/react-virtual` + 手动拍平
- 虚拟化后支持全量渲染 catalog，不再强依赖 UI 分页懒加载

### 4.5 Phase 4 前端测试

- 无列描述时 preview 正常渲染
- 有描述时 tooltip / 详情展示出现
- `attachedMetadata` 不被源 metadata 覆盖
— 参考 [测试策略 → 前端测试](./14-unified-source-metadata-plan.md#测试策略)

## Phase 5：Agent 数据理解与发现

> 参考：[实施计划 → Phase 5](./14-unified-source-metadata-plan.md#phase-5agent-数据理解与发现)

### 5.1 分析 Agent 摘要增强 ✅

- `generate_data_summary()` 同时使用 `attached_metadata` 和 `TableMetadata.description`
- `get_field_summary()` 存在 `ColumnInfo.description` 时输出列描述
- `inspect_source_data` 自动受益（无需额外改动，但需验证）
— 参考 [Agent 上下文 → 用途一](./14-unified-source-metadata-plan.md#用途一分析-agent-读取表列描述)

> 实现：`agent_utils.py` — `get_field_summary` 增加 `column_description` 参数；`generate_data_summary` 增加 `col_desc_cache`/`table_desc_cache` 从 `WorkspaceMetadata` 读取

### 5.2 Catalog 缓存机制 ✅

- 首次连接数据源时调用轻量化 `list_tables()` 全量拉取
- 结果持久化到 `<user_home>/catalog_cache/<source_id>.json`（用户级，跨 workspace 共享）
- 用户主动点击"刷新 catalog"时重新拉取覆盖磁盘文件
- 用户断开连接或删除连接器时删除对应缓存文件
— 参考 [Catalog 缓存机制](./14-unified-source-metadata-plan.md#catalog-缓存机制)

> 实现：新文件 `datalake/catalog_cache.py`（save/load/delete/search）；`data_connector.py` 在 `get-catalog-tree`/`disconnect`/`delete` 路由中集成

### 5.3 Workspace 本地搜索（第一层） ✅

- `WorkspaceMetadata` 上实现 `search_tables(query)` 方法
- 搜索 `workspace.yaml` 中表名、`description`、列名、`ColumnInfo.description`
— 参考 [搜索工具伪代码 → 第一层](./14-unified-source-metadata-plan.md#搜索工具伪代码)

> 实现：`workspace_metadata.py` — `WorkspaceMetadata.search_tables()` 方法，带评分和排序

### 5.4 磁盘 Catalog 缓存搜索（第二层） ✅

- 使用 Python 子串匹配查询 `catalog_cache/*.json`（DuckDB 方案可后续按需升级）
- 支持子串匹配和列级搜索
- 缓存文件不存在时跳过，不做远程 fallback
- 已导入表在缓存搜索结果中去重
— 参考 [Agent 如何拿到 catalog 缓存](./14-unified-source-metadata-plan.md#agent-如何拿到-catalog-缓存)

> 实现：`catalog_cache.py` — `search_catalog_cache()` 函数

### 5.5 search_data_tables Agent 工具 ✅

- 定义 `search_data_tables(query, scope)` 工具
- 整合两层搜索，返回去重排序后的 Level 0 摘要
- 排序策略：表名匹配 > 表描述匹配 > 列名/列描述匹配
- 结果数量限制：workspace 层 ≤ 50 条，每个数据源 ≤ 20 条
— 参考 [搜索工具伪代码](./14-unified-source-metadata-plan.md#搜索工具伪代码)

> 实现：`context.py` — `handle_search_data_tables()` handler；`data_agent.py` — 工具定义、系统提示更新、dispatch 逻辑

### 5.6 渐进式上下文策略 ✅

- 搜索结果默认 Level 0（表名 + 一行描述 + 匹配列名）
- `inspect_source_data` 支持 Level 1（schema 概览）和 Level 2（完整细节）
- 请求表数量 > 3 时自动降级为 Level 1，≤ 3 时返回 Level 2
— 参考 [大规模 Catalog 的存储分层与上下文管理 → 渐进式上下文策略](./14-unified-source-metadata-plan.md#大规模-catalog-的存储分层与上下文管理)

> 实现：`context.py` — `handle_inspect_source_data()` 根据 `len(tables_to_inspect)` 自动调整 `include_data_samples` 和 `char_limit`

### 5.7 Refresh 元数据同步 ✅

- Refresh 时重新拉取源 metadata
- 新 metadata 可用时更新 `TableMetadata.description`
- metadata 不可用时保留旧描述
- 源系统返回空描述时清空（以源为准）
- 不触碰 `attachedMetadata`
— 参考 [数据流 → Refresh](./14-unified-source-metadata-plan.md#refresh)

> 实现：Phase 1.4 已在 `connector_refresh_data` 中集成 `_merge_source_metadata`；Phase 5 修正了空描述清空逻辑（`"description" in source_meta` 而非 `source_meta.get("description")`）

### 5.8 安全边界 ✅

- workspace 搜索只能用 Agent 绑定的 `self.workspace`
- catalog 缓存搜索只能读当前用户 `catalog_cache/` 目录
- Agent 不能通过搜索工具建立新连接或触发远程拉取
- 搜索结果不包含 `loader_params`、凭证、连接串、内部路径
— 参考 [安全边界：Agent 工具的 workspace 隔离](./14-unified-source-metadata-plan.md#安全边界agent-工具的-workspace-隔离)

> 验证通过：handler 使用 `self.workspace` 和 `self.workspace.workspace_root`；catalog cache 仅存 name/metadata/columns；disconnect/delete 路由清理缓存
