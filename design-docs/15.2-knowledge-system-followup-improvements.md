# 15.2 知识系统后续改进项

> 前置文档：[15-agent-knowledge-reasoning-log.md](./15-agent-knowledge-reasoning-log.md)（设计）、[15.1-agent-knowledge-reasoning-log-dev-plan.md](./15.1-agent-knowledge-reasoning-log-dev-plan.md)（开发计划）
> 状态：待讨论
> 最后更新：2026-04-27

Phase 2–4 实施完成后，经逐项核对发现以下可改进项。它们**不影响功能完整性和安全性**，但会影响知识自动注入的实际效果和代码风格一致性。

---

## 1. 搜索策略：整串子串匹配 → 分词加权匹配

### 现状

`_search_relevant_knowledge` 把 `user_question`（如 "Show quarterly sales trend by region"）和 `table_names`（如 `["sales_data", "regions"]`）用空格拼成一个长字符串，直接传给 `KnowledgeStore.search()`。

`_match_score` 做的是**整个 query 作为一个子串**去匹配 title / tags / body：

```text
# 当前逻辑（伪代码）
q = "show quarterly sales trend by region sales_data regions"
if q in title.lower():     # 整句匹配 — 几乎不可能命中
    score += 100
```

只有当 title/tag 恰好**包含整句**时才能匹配。对于短 query（如 Agent 主动调用 `search_knowledge("ROI")`），效果没问题；但对于自动注入场景的长 query，命中率几乎为零。

### 设计文档的预期

设计文档 Part 2 "知识注入策略" 中写的是：

> ```text
> query_terms = extract_keywords(user_question, table_names)
> relevant = knowledge_store.search(query_terms, ...)
> ```

即应先提取关键词，再搜索。

### 改进方案

**方案 A：分词匹配（推荐）**

在 `_match_score` 层面支持多词匹配：

```text
# 改进后（伪代码）
tokens = tokenize(query)  # ["quarterly", "sales", "trend", "region"]

for token in tokens:
    if token in title.lower():
        score += 100 / len(tokens)  # 按词数平摊权重
    if any(token in tag.lower() for tag in tags):
        score += 50 / len(tokens)
    ...
```

- 优点：改动集中在 `_match_score` 一处，API 不变
- 分词策略：简单空格分词 + 过滤停用词（the, a, is, by, of, ...）+ 过滤过短词（≤2 字符）
- 保留整串匹配的加分项（如果 query 恰好完整命中 title，额外加分）
- 对中文 query 的处理：暂不分词，整串匹配（中文分词复杂度高，后续可引入 jieba）

**方案 B：调用端分词**

在 `_search_relevant_knowledge` 中先提取关键词，多次调用 `search()`，合并去重：

```text
keywords = extract_keywords(user_question)  # 提取 2-4 个核心词
for kw in keywords:
    results += knowledge_store.search(kw, ...)
deduplicate(results)
```

- 优点：不改 `KnowledgeStore`，搜索语义清晰
- 缺点：多次遍历文件系统，性能较差（条目多时 N × M）

**推荐方案 A**。改动最小，且 Agent 主动调用 `search_knowledge` 工具时也能受益。

### 影响范围

| 文件 | 改动 |
|---|---|
| `knowledge/store.py` | `_match_score` 增加分词逻辑 |
| `data_agent.py` | `_search_relevant_knowledge` 可简化 query 构建（不再需要拼接长字符串） |
| `test_knowledge_store.py` | 新增多词匹配、停用词过滤的测试 |
| `test_agent_knowledge_integration.py` | 加强 `test_relevant_knowledge_injected` 断言 |

## 2. 测试断言加强：`test_relevant_knowledge_injected`

### 现状

```text
assert "[RELEVANT KNOWLEDGE]" in user_msg or agent._injected_knowledge == []
```

`or` 条件使得无论搜索是否命中，测试都能通过。这是为了绕过搜索策略问题的妥协。

### 改进

修复搜索策略（第 1 项）后，测试数据中的 skill title "Handle Missing Values" 应能被 query "How to handle missing values?" 中的分词 `"missing"` 或 `"values"` 命中。届时应改为：

```text
assert "[RELEVANT KNOWLEDGE]" in user_msg
assert len(agent._injected_knowledge) >= 1
assert any("Handle Missing Values" in k["title"] for k in agent._injected_knowledge)
```

**依赖**：必须先完成第 1 项搜索策略改进。

## 3. `list_all` / `search` 中文件读取风格统一

### 现状

`list_all` 和 `search` 通过 `jail.rglob("*.md")` 获取文件路径列表（路径保证在 jail 内），但随后用 `md_file.read_text(encoding="utf-8")` 直接读取，绕过了 `jail.read_text()`。

```text
for md_file in jail.rglob("*.md"):        # ← 通过 ConfinedDir API
    raw = md_file.read_text(encoding="utf-8")  # ← 绕过 ConfinedDir，直接 Path.read_text
```

**安全影响**：无。`rglob` 返回的绝对路径已经在 jail 根目录下，不存在逃逸可能。

**风格影响**：与 `read` / `write` / `delete` 方法统一通过 `jail.read_text()` / `jail.write_text()` 访问的模式不一致。后续新增开发者可能产生误解（"为什么这里可以直接用 Path？"）。

### 改进方案

把 `md_file.read_text()` 改为通过 `jail.read_text(rel_path)` 访问：

```text
for md_file in jail.rglob("*.md"):
    rel = md_file.relative_to(jail.root)            # 计算相对路径
    raw = jail.read_text(str(rel).replace("\\", "/"))  # 通过 ConfinedDir 读取
```

**代价**：每次 `read_text` 内部会调用 `resolve()`，增加一次路径校验。对于知识库规模（预期几十到上百条），性能影响可忽略。

### 影响范围

| 文件 | 改动 |
|---|---|
| `knowledge/store.py` | `list_all` 和 `search` 两处 `md_file.read_text()` → `jail.read_text(rel)` |

## 4. "保存为经验"按钮位置与 experience_context 边界

### 现状

前端在 `DataThreadCards.tsx` 中根据 derived table 的 `trigger.interaction` 是否包含用户消息来显示"保存为经验"按钮：

```text
table.derive.trigger.interaction?.some(e => e.from === "user")
```

DataAgent 多步生成 table 时，第一张 table 通常携带初始用户 prompt；生成后续 table 时新的 draft interaction 会被重置。因此按钮容易出现在第一个 AI 生成的 table 上，而不是当前分析链最终结果上。

### 设计文档的预期

设计文档 Part 4 "经验保存入口" 已明确：

- 按钮只出现在当前分析链最后一个 leaf table 上
- 中间 table 不直接显示该按钮
- `experience_context` 从触发保存的 leaf table 回溯当前仍存在的 derived chain
- 用户已删除的中间 table 及其 table-level step 不进入经验
- 最终 leaf table 仍可保存为经验；属于最终成功过程的失败/修复摘要保留

### 改进方案

**方案 A：渲染时判断 leaf table（推荐）**

在构建 table card 时增加判断：

```text
isLeaf = !tables.some(t => t.derive?.trigger.tableId === table.id && !t.anchored)
```

只有 `table.derive` 存在且 `isLeaf` 为 true 时才显示"保存为经验"按钮。保存时从 leaf table 向上回溯当前 Redux 中仍存在的 chain，构建 `experience_context`。

- 优点：符合用户语义，改动集中在前端
- 删除中间 table 后，当前 Redux chain 已经重挂到祖先 table，可自然排除已删除节点
- 需要确保按钮不出现在 anchored table 或非 DataAgent 生成的无用户上下文 table 上

**方案 B：保存按钮仍按当前条件显示，但点击时定位 leaf table**

点击任意带用户 prompt 的 table 时，自动寻找该分析链的 leaf table 并保存 leaf 上下文。

- 优点：用户可从早期节点触发保存
- 缺点：UI 表达不清晰，用户点击第一个 table 却保存最后一个 table，容易造成误解

**推荐方案 A**。按钮位置本身应表达保存范围，避免用户误以为保存的是中间步骤。

### 影响范围

| 文件 | 改动 |
|---|---|
| `DataThreadCards.tsx` | 传入 `tables` 判断 leaf table；仅 leaf table 渲染 `SaveExperienceButton` |
| `SaveExperienceButton.tsx` | 构建 `experience_context` 时支持从 leaf table 回溯当前可见链，排除已删除节点 |
| 前端测试/手动验证 | 覆盖多步分析、删除中间 table、失败/修复摘要保留 |

## 优先级建议

| 项目 | 优先级 | 理由 |
|---|---|---|
| 搜索策略改进 | **高** | 直接影响 Phase 3.2 "按需搜索注入" 的实际效果，当前形同虚设 |
| "保存为经验"按钮位置 | **高** | 当前按钮出现在第一个中间 table，和"保存最终满意分析"语义不一致 |
| 测试断言加强 | **中** | 依赖搜索策略修复，修复后顺带完成 |
| 文件读取风格统一 | **低** | 纯风格问题，无安全/功能影响 |
