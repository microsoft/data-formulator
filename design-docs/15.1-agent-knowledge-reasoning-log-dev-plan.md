Phase 1 和 Phase 2 都依赖扩展后的 `ConfinedDir`，因此先完成此步。

### 0.1 ConfinedDir 新增 API

- `read_text(relative, encoding="utf-8") -> str` — 安全读取文本文件
- `write_text(relative, content, encoding="utf-8") -> Path` — 安全写入文本（自动创建父目录）
- `exists(relative) -> bool` — 安全检查文件/目录是否存在（路径逃逸返回 False）
- `iterdir(relative="") -> Iterator[Path]` — 列出子目录内容
- `rglob(pattern, relative="") -> Iterator[Path]` — 递归搜索文件
- `unlink(relative) -> None` — 安全删除文件
- 所有新方法内部均通过 `self.resolve()` 做路径约束检查
— 参考 [ConfinedDir 需要增强的 API](./15-agent-knowledge-reasoning-log.md#confineddir-需要增强的-api)

## Phase 1：推理日志（Reasoning Log）

> 参考：[Part 1：推理日志](./15-agent-knowledge-reasoning-log.md#part-1推理日志reasoning-log)

### 1.1 ReasoningLogger 类实现

> 文件：`py-src/data_formulator/agents/reasoning_log.py`（新建）

- `ReasoningLogger.__init__(identity_id, agent_type, session_id)` — 创建日志文件到 `DATA_FORMULATOR_HOME/agent-logs/<date>/<safe_identity_id>/<session_id>-<agent_type>.jsonl`
- 内部通过 `ConfinedDir(get_data_formulator_home() / "agent-logs" / today_str() / safe_identity_id)` 约束路径
- `log(step_type, **kwargs)` — 写入一行 JSON，包含 `step_type` + `ts` + kwargs
- `close()` — 关闭文件描述符
- 实现上下文管理器 `__enter__` / `__exit__`，确保异常时也能 close
- `ts` 字段使用 UTC ISO 格式
- 每次 `log()` 后 `flush()`，确保实时可读
— 参考 [实现方式](./15-agent-knowledge-reasoning-log.md#实现方式)

### 1.2 环境变量开关 DF_AGENT_LOG

- 读取 `os.environ.get("DF_AGENT_LOG", "on")`，支持三档：`off` / `on` / `verbose`
- `off` 模式：`log()` 为 no-op，不创建文件，不产生 IO
- `on` 模式：写结构化摘要（messages_count、token 估算、tool 名等），**不**记录完整 messages 内容
- `verbose` 模式：记录完整 messages 内容，**必须**经过 `log_sanitizer` 清洗后写入
- 环境变量值大小写不敏感（`OFF` / `On` / `VERBOSE` 均有效）
— 参考 [开关与级别](./15-agent-knowledge-reasoning-log.md#开关与级别)

### 1.3 过期日志清理

- Agent 启动时 best-effort 清理 30 天前的日志目录
- 清理在后台执行，**不阻塞**请求处理
- 清理失败仅 `logger.warning()`，不抛异常
- 清理逻辑只删除 `agent-logs/` 下的日期子目录，不删除其他内容
— 参考 [开关与级别](./15-agent-knowledge-reasoning-log.md#开关与级别)（"保留天数硬编码 30 天"）

### 1.4 安全约束

- `on` 模式下不记录用户数据原文、完整 messages 内容
- `verbose` 模式下完整内容经 `log_sanitizer.sanitize_params()` 清洗后写入
- 日志中不出现 API key、凭证、连接串
- 日志文件通过 `ConfinedDir` 约束到系统级管理员日志目录下
- 编写安全测试验证 verbose 模式的清洗效果
— 参考 [安全约束](./15-agent-knowledge-reasoning-log.md#安全约束)

### 1.5 DataAgent 集成埋点

> 文件：`py-src/data_formulator/agents/data_agent.py`

- `DataAgent.__init`__ 或 `run()` 入口 → 创建 `ReasoningLogger` 实例
- `session_start` — user_question, input_tables, model, rules_injected
- `context_built` — system_prompt_tokens, user_msg_tokens, total_tables, primary_tables
- `llm_request` — iteration, messages_count, tools_available
- `llm_response` — iteration, latency_ms, finish_reason, tool_calls 或 action 摘要
- `tool_execution` — tool 名, 输入摘要, 输出摘要, latency_ms
- `action_execution` — action 类型, status, output_rows, chart_type
- `repair_attempt` — attempt 次数, 原始错误, 修复后 status
- `session_end` — status, total_iterations, total_llm_calls, total_latency_ms
- `run()` 末尾 `finally` 块中调用 `reasoning_log.close()`
— 参考 [关键字段说明](./15-agent-knowledge-reasoning-log.md#关键字段说明) 和 [实现方式](./15-agent-knowledge-reasoning-log.md#实现方式)

## Phase 2：知识存储后端（KnowledgeStore）

> 参考：[Part 2：知识系统](./15-agent-knowledge-reasoning-log.md#part-2知识系统knowledge-base)

### 2.1 KnowledgeStore 类实现

> 文件：`py-src/data_formulator/knowledge/store.py`（新建）

- `__init__(user_home: Path)` — 创建 `ConfinedDir` 实例管理 rules / skills / experiences 三个子目录
- `validate_path(category, relative_path)` — 校验路径深度：rules 只允许文件（1 层），skills/experiences 最多 2 层
- 校验文件必须是 `.md` 格式
- 路径穿越（`../`）由 `ConfinedDir` 拦截
— 参考 [目录结构约束](./15-agent-knowledge-reasoning-log.md#目录结构约束)

### 2.2 CRUD 操作

- `list_all(category) -> list[dict]` — 列出该分类下所有条目（解析 front matter 的 title、tags、source）
- `read(category, path) -> str` — 读取知识文件完整内容
- `write(category, path, content) -> Path` — 创建或更新知识文件（自动 front matter 验证）
- `delete(category, path) -> None` — 删除知识文件
- CRUD 均通过 `ConfinedDir` 读写，不直接使用 `Path` 拼接
— 参考 [经验管理 API](./15-agent-knowledge-reasoning-log.md#经验管理-api)

### 2.3 Front Matter 解析

- 实现 `parse_front_matter(content: str) -> tuple[dict, str]` — 解析 YAML front matter 和正文
- 支持 `title`（必需）、`tags`（可选）、`created`（必需）、`updated`（必需）、`source`（可选）、`source_context`（可选）、`relevance`（可选）
- front matter 格式不正确时返回空 dict + 全文作为 body（优雅降级）
— 参考 [文件格式](./15-agent-knowledge-reasoning-log.md#文件格式)

### 2.4 知识搜索

- `search(query, categories, max_results=10) -> list[dict]` — 跨分类搜索
- 搜索目标：title、tags、文件名（stem）、正文前 200 字
- 子串匹配（大小写不敏感）
- 返回字段：category、title、tags、path、snippet（前 500 字）、source
- 按匹配度排序（title 匹配 > tags 匹配 > 文件名匹配 > 正文匹配）
- 限制返回数量
— 参考 [知识搜索](./15-agent-knowledge-reasoning-log.md#知识搜索)

### 2.5 Knowledge API 路由

> 文件：`py-src/data_formulator/routes/knowledge.py`（新建）

- `POST /api/knowledge/list` — `{ category }` → 列出该分类下所有条目
- `POST /api/knowledge/read` — `{ category, path }` → 读取完整内容
- `POST /api/knowledge/write` — `{ category, path, content }` → 创建/更新
- `POST /api/knowledge/delete` — `{ category, path }` → 删除
- `POST /api/knowledge/search` — `{ query, categories? }` → 搜索
- 所有 API 通过 `get_identity_id()` 确定用户
- 所有 API 通过 `ConfinedDir` 约束访问范围
- 错误处理遵循统一错误协议（HTTP 200 + `status: "error"`）
- 注册 Blueprint 到 Flask app
— 参考 [经验管理 API](./15-agent-knowledge-reasoning-log.md#经验管理-api)

## Phase 4：经验提炼（Experience Distillation）

> 参考：[Part 3：经验提炼](./15-agent-knowledge-reasoning-log.md#part-3经验提炼experience-distillation)

### 4.1 ExperienceDistillAgent 实现

> 文件：`py-src/data_formulator/agents/agent_experience_distill.py`（新建）

- `ExperienceDistillAgent` 类，输入前端 `experience_context`，输出 Markdown 经验文档
- `SYSTEM_PROMPT` 指导 LLM 提炼：标题、场景、方法、要点、标签、失败/修复路径、用户澄清如何改变方向
- `_extract_context_summary(experience_context)` — 提取 user_question、interaction、result_summary、execution_attempts、dialog 摘要；
`execution_attempts` 包含失败/修复代码的结构化摘要（如操作类型、非空行数），不传递代码原文
- 输出格式为 Markdown with YAML front matter（`source: agent_summarized`，`source_context: <context_id>`）
- LLM 调用遵循现有 client 模式
— 参考 [经验提炼 Agent](./15-agent-knowledge-reasoning-log.md#经验提炼-agent)

### 4.2 经验提炼 API

- `POST /api/knowledge/distill-experience` — `{ experience_context, model, category_hint? }`
- 从前端提交的当前用户可见分析上下文提炼，不读取管理员推理日志
- 调用 `ExperienceDistillAgent.run_from_context()` 提炼
- 结果写入 `knowledge/experiences/` 目录（如有 category_hint 则写入对应子目录）
- 返回新创建的经验条目信息
- 错误处理：缺少 `experience_context` 或必需字段时返回友好错误
- 遵循统一错误协议
— 参考 [流程](./15-agent-knowledge-reasoning-log.md#流程) 和 [经验管理 API](./15-agent-knowledge-reasoning-log.md#经验管理-api)

### 6.6 "保存为经验"按钮

- DataAgent 完成分析（`present` action）后，只在当前分析链最后一个 leaf table 的结果卡片上显示"保存为经验"按钮
- leaf table 定义：当前仍存在、且没有其他未锚定 derived table 继续从它派生的最终结果 table
- 中间 table 不直接显示"保存为经验"按钮，避免把半成品步骤保存为完整经验
- 点击弹出确认对话框（可选子目录、可编辑标题）
- 确认后调用 `distillExperience` thunk，提交该 leaf table 对应的 `experience_context`
- `experience_context` 从触发保存的 leaf table 回溯当前仍存在的 derived chain，只收集当前可见链上的 `dialog`、`interaction`、`result_summary`、`execution_attempts`
- 如果用户删除了中间 table，保存经验时不包含已删除节点及其 table-level step；最终 leaf table 仍可保存为经验
- 属于最终 leaf table 成功过程的失败/修复摘要保留；属于已删除中间 table 的失败/修复摘要排除
- 完成后知识面板刷新显示新条目
- 按钮文本和对话框文本通过 `t()` i18n 国际化
— 参考 [经验保存入口](./15-agent-knowledge-reasoning-log.md#经验保存入口)

## 横切关注点

以下事项贯穿所有 Phase，需持续关注：

### 错误处理

- 所有 API 端点遵循统一错误协议：HTTP 200 + `status: "error"` + `error: {code, message, retry}`
- 新增 `ErrorCode` 常量（如需要）需同步添加到 `errors.py`、`errorCodes.ts`、`en/errors.json`、`zh/errors.json`
- 日志不包含凭证、token、连接串或敏感路径
— 参考 `dev-guides/7-unified-error-handling.md`、`.cursor/rules/error-response-safety.mdc`

### 路径安全

- 所有知识文件/推理日志的读写通过 `ConfinedDir` 约束
- 禁止裸 `Path(root) / user_input` 拼接
- 新增模块必须遵循 `dev-guides/8-path-safety.md`  
— 参考 `.cursor/skills/path-safety/SKILL.md`

### 日志脱敏

- 推理日志 verbose 模式必须经过 `log_sanitizer` 清洗
- 新增的 `logger.*()` 调用不得记录 API key、token、连接串
— 参考 `dev-guides/2-log-sanitization.md`、`.cursor/rules/log-sanitization.mdc`

### 国际化（i18n）

- 前端所有用户可见文本通过 `t()` 调用
- 后端 Agent 消息通过 `message_code` 机制翻译
