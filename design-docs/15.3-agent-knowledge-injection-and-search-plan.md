# 15.3 Agent 知识注入与搜索工具规划

> 状态：规划中  
> 最后更新：2026-05-01  
> 前置文档：`15-agent-knowledge-reasoning-log.md`、`15.1-agent-knowledge-reasoning-log-dev-plan.md`、`15.2-knowledge-system-followup-improvements.md`  
> 开发规范：`dev-guides/10-agent-knowledge-reasoning-log.md`

## 1. 结论

当前知识系统已经接入一部分 Agent，但还需要在下一轮规划中把“哪些 Agent 需要哪类知识”固化为明确矩阵，并把知识搜索从各 Agent 的零散调用收敛成一个共享能力。

核心结论：

1. `rules` 是约束类知识。`alwaysApply != false` 的 rules 应注入所有需要遵守用户规则的 Agent 的 system prompt；`alwaysApply: false` 的 rules 只应通过相关性搜索注入。
2. `skills` 是做法类知识。它们不应全量注入，应按当前任务搜索摘要后注入，避免污染 prompt。
3. `experiences` 是案例类知识。它们也不应全量注入，应按问题、表名、图表字段或分析上下文搜索摘要后注入。
4. 当前 `DataAgent` 已实现 `search_knowledge` / `read_knowledge` 内部工具，但底层是 `KnowledgeStore.search()` 的 Markdown 子串搜索，不是 DuckDB。
5. DuckDB 搜索目前用于 `search_data_tables` 的 catalog cache 表目录搜索，不用于 knowledge 搜索。
6. 知识搜索现阶段不需要 LLM 子 Agent。应先实现确定性的后端搜索服务，返回安全摘要；只有未来需要跨多篇知识文档做语义综合时，才考虑引入 summarizer 子 Agent。

## 3. 当前知识搜索实现

### 3.1 已实现部分

当前已经有两个层次的知识搜索能力：

1. `KnowledgeStore.search(query, categories, max_results)`  
   位置：`py-src/data_formulator/knowledge/store.py`。  
   搜索 `rules`、`skills`、`experiences` 下的 Markdown 文件，返回 `category`、`title`、`tags`、`path`、`snippet`、`source`。

2. `DataAgent` 内部工具 `search_knowledge` / `read_knowledge`  
   位置：`py-src/data_formulator/agents/data_agent.py`。  
   `search_knowledge` 返回摘要列表，`read_knowledge` 根据 category/path 读取完整 Markdown。工具错误会以短文本返回给 LLM，并在服务端日志记录。

当前 `DataAgent` 自动注入也复用 `KnowledgeStore.search()`：

```text
user_question + table_names
  -> KnowledgeStore.search(categories=["rules", "skills", "experiences"])
  -> [RELEVANT KNOWLEDGE] 摘要块
```

### 3.2 当前限制

当前搜索不是 DuckDB，也不是语义搜索。它是大小写不敏感的整串子串匹配：

```text
query in title      -> +100
query in tags       -> +50
query in filename   -> +30
query in body prefix -> +10
```

这对短 query 有效，例如 `ROI`；但对自动注入场景效果弱，例如：

```text
"show quarterly sales trend by region sales_data regions"
```

整句很难完整出现在 title/tag/body 中，因此经常无命中。这个问题已在 `15.2` 中列为待改进项。

### 3.3 DuckDB 搜索现状

DuckDB 目前存在于数据表目录搜索，不在知识库搜索：

| 工具/函数 | 位置 | 搜索对象 | 是否 DuckDB | 说明 |
|---|---|---|---|---|
| `search_knowledge` | `DataAgent` 工具 | `knowledge/rules|skills|experiences/*.md` | 否 | 调用 `KnowledgeStore.search()`，返回知识摘要 |
| `read_knowledge` | `DataAgent` 工具 | 单个知识 Markdown | 否 | 读取完整知识正文 |
| `search_data_tables` | `DataAgent` 工具 | workspace metadata + catalog cache | 间接是 | 调用 `handle_search_data_tables()` |
| `search_catalog_cache` | `datalake/catalog_cache.py` | `<user_home>/catalog_cache/*.json` | 是 | 默认 `_search_duckdb()`，失败回退 Python |
| `read_catalog_metadata` | `DataAgent` 工具 | catalog cache 单表详情 | 否 | 读取并合并用户 annotations |

也就是说，“DuckDB 实现类似 find/group 的搜索”目前更接近 catalog cache 的表发现能力：从缓存 JSON 中查找表名、描述、列名、列描述，并返回分组后的轻量摘要。它不是 knowledge rules/skills/experiences 的搜索实现。

## 4. 搜索工具应如何规划

### 4.1 目标形态

建议把知识搜索抽象为共享的 `KnowledgeSearchService`，所有需要知识注入的 Agent 都通过它，而不是各自拼 query、各自格式化摘要。

使用方式应是“同一个搜索工具，不同 category scope”：

| 知识类型 | 调用方式 | 注入方式 |
|---|---|---|
| always rules | `load_always_rules()` | 每次注入需要 rules 的 Agent system prompt |
| non-always rules | `search_relevant(categories=["rules"])` | 相关时摘要注入 |
| skills | `search_relevant(categories=["skills"])` | 相关时摘要注入 |
| experiences | `search_relevant(categories=["experiences"])` | 相关时摘要注入 |
| 混合知识 | `search_relevant(categories=["rules", "skills", "experiences"])` | 仅适合 `DataAgent` 这类综合分析 Agent |

建议接口：

```python
class KnowledgeSearchService:
    def load_always_rules(self) -> list[KnowledgeHit]:
        ...

    def search_relevant(
        self,
        query: str,
        categories: list[str],
        *,
        exclude_always_rules: bool = True,
        max_results: int = 5,
    ) -> list[KnowledgeHit]:
        ...

    def format_summary_block(
        self,
        hits: list[KnowledgeHit],
        heading: str,
    ) -> str:
        ...
```

`KnowledgeHit` 应至少包含：

| 字段 | 说明 |
|---|---|
| `category` | `rules` / `skills` / `experiences` |
| `title` | front matter 标题 |
| `path` | category 内相对路径 |
| `snippet` | 安全摘要，默认不返回全文 |
| `tags` | 搜索标签 |
| `score` | 排序分数，便于调试和测试 |
| `alwaysApply` | rules 专用，用于避免重复注入 |

### 4.2 搜索策略

本轮不引入 LLM 子 Agent 或复杂向量检索，直接实现 DuckDB-backed 确定性搜索：

1. 分词匹配  
   对英文 query 做简单 tokenization，过滤停用词和过短词；中文 query 暂保留整串匹配。

2. 保留整串加分  
   如果 query 完整命中 title/tag/body，额外加分，兼容短 query。

3. 分类权重  
   `rules` 的 title/tag 命中权重更高；`skills` 和 `experiences` 的 body/snippet 命中可以稍高。

4. always rules 去重  
   `load_always_rules()` 已全量注入的 rules，不应再次出现在相关知识摘要中，除非调用方显式要求。

5. 输出摘要，不输出全文  
   自动注入只放 snippet；LLM 需要全文时由 `read_knowledge` 明确读取。

### 4.3 是否需要 DuckDB

本轮直接上 DuckDB 知识索引。DuckDB 可以让 knowledge 搜索变得更复杂、更稳定，但前提是先把 Markdown 知识文件解析成结构化索引。DuckDB 本身不会自动让搜索“更语义化”；它提升的是结构化过滤、字段加权、分组、排序和可测试性。

本轮 DuckDB 搜索应实现：

1. 字段加权搜索：`title`、`tags`、`description`、`body_prefix`、`source` 分别计分，而不是简单整串子串匹配。
2. 多词 query 匹配：把 query 拆成 tokens 后，用 SQL 对每个 token 计算命中字段和分数。
3. category scope：同一个工具按 `rules`、`skills`、`experiences` 或混合范围搜索。
4. always rules 过滤：搜索 non-always rules 时可以直接 `WHERE always_apply = false`，避免和 system prompt 全量注入重复。
5. find/grep/group 类语义：先 find 候选文档，再 grep 正文/摘要命中片段，最后按 category、tag、source、folder 分组，每组限制返回数量，避免某一类知识淹没结果。
6. 更稳定排序：综合 `field_weight + token_coverage + recency + source_weight`，比 Python 当前的 4 条 if 更可控。
7. 调试可见性：返回 `score`、`match_fields`、`matched_tokens`，方便测试为什么命中。

实现方式：

1. `KnowledgeStore` 继续负责 Markdown 文件 CRUD、front matter 解析和路径安全。
2. 新增只读 `knowledge_index` 派生索引，索引由 `KnowledgeStore` 或 `KnowledgeSearchService` 从 Markdown 构建。
3. `KnowledgeSearchService.search_relevant()` 默认走 DuckDB 查询索引；DuckDB 初始化失败时可以降级为安全的 Python fallback，但主路径和测试应以 DuckDB 为准。
4. `read_knowledge` 仍通过 `KnowledgeStore.read()` 读取全文，不从 DuckDB 读取正文。

索引字段：

```text
knowledge_index
  category
  path
  title
  description
  tags_text
  source
  body_text
  body_prefix
  always_apply
  created_at
  updated_at
```

DuckDB 的目标不是替代 `KnowledgeStore` 的文件读写，而是作为只读检索层，为 Agent 返回更精准、更可解释的摘要结果。

### 4.4 与 find/grep/group 类搜索语义的关系

DuckDB 知识索引可以参考 catalog cache 的思路，但语义应保持简单：

1. `find` 类能力：按 category、path、filename、title、tags、source 等结构化字段找候选文档。
2. `grep` 类能力：按 query tokens 在 `description`、`body_prefix`、必要时 `body_text` 中匹配内容，并返回命中字段、命中 token 和短 snippet。
3. `group` 类能力：按 category、tag、source 或 path folder 聚合，限制每组返回数量。
4. `summary` 类能力：只返回 title/path/snippet/score/match_fields/matched_tokens，不默认返回全文。

这仍然应是后端确定性服务，而不是 LLM 子 Agent。LLM 只消费搜索结果；如果需要全文，再显式调用 `read_knowledge`。

### 4.5 是否需要子 Agent

当前不建议引入子 Agent 执行搜索。

原因：

1. 搜索是确定性检索任务，后端服务更容易测试、复现和控制安全边界。
2. 子 Agent 会增加 LLM 调用成本和延迟。
3. 子 Agent 容易把“检索摘要”和“解释推断”混在一起，增加 hallucination 风险。
4. 当前 Agent 已可以通过 `read_knowledge` 对少数命中文档取全文，无需另起 Agent。

可以考虑子 Agent 的未来场景：

1. 用户知识库很大，命中十几篇经验，需要压缩成一个任务相关 brief。
2. 需要跨多个 experiences 做冲突合并、归纳模式或生成新的 meta-skill。
3. 需要语义理解而不是关键词匹配，并且已有明确的质量评估集。

即使未来引入，也建议命名为 `KnowledgeBriefAgent`，只接收后端确定性搜索返回的 top N 文档摘要或全文，不直接访问文件系统。

### 4.6 可借鉴的搜索与上下文调度设计

从类似 Agent 工具系统中可以借鉴以下 5 点：

1. 搜索结果必须有硬上限和分页：返回 `limit`、`offset`、`truncated`、`next_offset`，并限制 snippet 字符数和总返回字符数。
2. 同一套搜索支持多种输出模式：`find` 返回条目列表，`grep` 返回命中 snippet，`group` 返回聚合统计，`summary` 返回可直接注入 Agent 的精简摘要。
3. 只读任务可以并行，写任务必须串行：DuckDB search/read 是只读可并发；knowledge write/delete/distill 应串行或加版本保护。
4. 摘要和压缩要预留输出预算：如果需要把搜索结果压缩成 brief，必须先扣掉 summary 生成需要的 token，而不是把上下文塞满后再总结。
5. 检索和总结分层：基础搜索由 DuckDB 确定性完成；只有 top N 结果仍然过长或需要跨文档综合时，才交给 `KnowledgeBriefAgent` 做压缩摘要。

### 4.7 KnowledgeOrchestratorAgent 任务编排

本轮应规划一个 `KnowledgeOrchestratorAgent`（也可以先实现为 `KnowledgeOrchestrator` 服务，后续再升级为 Agent），负责复杂知识注入流程的任务编排。它不直接替代 DuckDB 搜索，而是决定哪些搜索可以并行、哪些写操作必须串行，以及如何在预算内合并结果。

建议职责：

| 职责 | 说明 |
|---|---|
| 任务拆分 | 根据调用方 Agent 声明的 categories 和 query source，拆出 rules、skills、experiences、mixed search 等子任务 |
| 并行调度 | 对只读 `find` / `grep` / `group` / `read` 任务并发执行，设置全局并发上限 |
| 串行保护 | 对 `write`、`delete`、`distill-experience`、索引刷新等写任务串行执行或使用版本锁 |
| 预算控制 | 对每个任务设置 `max_results`、`max_snippet_chars`、`max_total_chars`，并在返回中标记截断 |
| 结果合并 | 去重 always rules、合并多 category 结果、按 score/category/source 排序，并生成 Agent 可注入摘要 |
| 降级处理 | DuckDB 不可用时走 Python fallback；单个子任务失败时保留其他结果并记录 warning |

建议输入：

```python
KnowledgeSearchPlan(
    caller_agent="DataAgent",
    query_sources={
        "user_question": "...",
        "table_names": [...],
        "thread_summary": "...",
    },
    tasks=[
        {"mode": "find", "categories": ["rules"], "exclude_always": True},
        {"mode": "grep", "categories": ["skills"], "max_results": 3},
        {"mode": "grep", "categories": ["experiences"], "max_results": 3},
        {"mode": "group", "categories": ["experiences"], "group_by": "tag"},
    ],
    budget={"max_total_chars": 6000, "max_items": 8},
)
```

建议输出：

```python
KnowledgeSearchBundle(
    injected_rules=[...],
    search_results=[...],
    summary_block="...",
    diagnostics={
        "truncated": False,
        "tasks": [...],
        "elapsed_ms": 42,
    },
)
```

调度原则：

1. 只读 DuckDB 查询默认并行，但受全局并发上限控制。
2. 同一 category 的多个 query 可以合并成一次 SQL，减少重复扫描。
3. 写知识和刷新索引不能与同一用户的搜索索引构建交叉执行，避免读到半更新状态。
4. 编排器只返回结构化摘要；是否把摘要注入 system prompt 或 user context 由调用方 Agent 决定。
5. 如果未来启用 `KnowledgeBriefAgent`，它只作为编排器的最后一步压缩器，输入是确定性搜索结果，不直接访问文件系统。

### 4.8 未来规划：上下文压缩与自动记忆

未来当 Agent 开始接入更多知识、工具结果和多轮分析上下文后，需要一套独立的上下文压缩与记忆处理机制。该机制不属于本轮 DuckDB 搜索交付，但应在架构上预留接口。

#### 触发阈值

以 200K context window 为基准，建议达到 80% 时开始压缩：

```text
context_window = 200_000 tokens
compact_threshold = 160_000 tokens
```

触发压缩时不应等到上下文完全塞满。需要预留摘要生成本身的 output token，以及后续工具结果和模型回复的 buffer。

#### 压缩 Agent 分工

| Agent / 服务 | 触发条件 | 输入 | 输出 | 说明 |
|---|---|---|---|---|
| `ContextCompressionAgent` | conversation / tool context 达到 80% 阈值 | 对话、工具结果、已注入知识、当前任务状态 | 当前任务可继续使用的 compact context | 压缩工作上下文，保留目标、约束、关键发现、未完成事项和必要引用 |
| `KnowledgeBriefAgent` | 搜索命中的 experiences / skills 太多，超过注入预算 | DuckDB top N 搜索结果和 snippets | 与本次任务最相关的 brief | 只压缩确定性搜索结果，不直接访问文件系统 |
| `MemoryCuratorAgent` | 会话完成、用户显式确认、或高价值分析链出现 | 分析过程、结果、用户反馈、执行修复摘要 | 候选 memory patch | 自动提出要保存/更新的经验、skills 或 rules，但写入策略要分级 |
| `SkillDraftAgent` | 多次相似经验、重复操作、稳定修复模式出现 | experiences、执行摘要、用户反馈、工具成功路径 | skill 草稿 | 自动编写可复用做法，但默认进入 pending/待确认状态 |
| `RuleDraftAgent` | 用户反复表达稳定偏好、业务口径或强约束 | 用户反馈、反复修正、已确认 skills/experiences | rule 候选 | 只生成候选 rules；不静默写入，更不静默创建 `alwaysApply: true` |

#### 自动记忆处理建议

未来可以从“人工保存经验”升级为“自动记忆系统”，但建议按风险分层：

1. `experiences`：最适合自动更新。它记录成功分析模式、数据处理经验、业务案例和错误修复经验，风险相对低。
2. `skills`：可以半自动更新。适合沉淀稳定的数据处理方法、连接器使用技巧、分析套路，但应避免把一次性案例误写成通用技能。
3. `rules`：不建议完全自动写入。rules 是强约束，会影响后续所有相关 Agent；应只生成候选规则，经过用户确认后再保存，尤其是 `alwaysApply: true` rules。

#### 自动编写 rules / skills 的 Agent 建议

越来越多用户不会手写 `rules` 和 `skills`，因此未来应增加自动草稿能力，但不能把所有草稿都直接变成生效知识。

| Agent | 职责 | 默认落库策略 |
|---|---|---|
| `ExperienceDistillAgent` | 从一次成功分析链提炼经验 | 已有；可继续支持自动保存到 `experiences` |
| `SkillDraftAgent` | 从多次相似 experiences、重复清洗/转换步骤、修复模式中提炼通用技能 | 生成 skill 草稿；默认 pending，用户确认后写入 `skills` |
| `RuleDraftAgent` | 从反复出现的用户偏好、业务口径、约束性反馈中提炼规则 | 只生成 rule 候选；必须用户确认后写入 `rules` |
| `MemoryCuratorAgent` | 统一判断本轮会话是否值得更新 memory，并分发给 experience/skill/rule 草稿 Agent | 生成 memory patch，不直接绕过分级写入策略 |

建议判断标准：

1. 单次任务成功结果优先沉淀为 `experiences`。
2. 多次任务中重复出现、跨数据集可复用的方法，才提升为 `skills`。
3. 用户明确表达“以后都这样”“这个口径固定”“不要再做某类操作”时，才生成 `rules` 候选。
4. `rules.alwaysApply: true` 必须由用户确认，不能由模型静默决定。
5. 自动草稿应带来源引用、置信度和推荐理由，方便用户快速接受、编辑或拒绝。

建议默认策略：

| 记忆类型 | 自动生成 | 自动写入 | 用户确认 |
|---|---|---|---|
| `experiences` | ✅ | ✅ 可配置 | 可选 |
| `skills` | ✅ | ⚠️ 建议半自动 | 建议确认 |
| `rules` | ✅ 候选 | ❌ 默认不自动写入 | 必须确认 |

#### 与任务编排的关系

`KnowledgeOrchestratorAgent` 未来应负责调度这些压缩/记忆任务：

1. 先执行 DuckDB `find` / `grep` / `group`，拿到候选知识。
2. 如果候选结果超过预算，调用 `KnowledgeBriefAgent` 压缩为任务相关 brief。
3. 如果整体上下文超过 160K tokens，调用 `ContextCompressionAgent` 压缩会话上下文。
4. 会话结束后，调用 `MemoryCuratorAgent` 判断是否需要生成 experience、skill 或 rule 候选。
5. `MemoryCuratorAgent` 可调用 `ExperienceDistillAgent`、`SkillDraftAgent`、`RuleDraftAgent` 生成分类型 memory patch。
6. 写入 memory 时继续遵守只读并行、写入串行原则；rules 写入必须经过确认。

#### 边界约束

1. 压缩结果必须保留来源引用，例如 knowledge `category/path`、table name、analysis step id。
2. 自动记忆不能保存凭证、连接串、token、原始大数据样本或敏感日志。
3. rules 的 `alwaysApply: true` 不应由模型静默创建。
4. 压缩 Agent 不能改变事实，只能压缩和重组已存在上下文。
5. 所有自动 memory patch 应记录来源 session、触发原因和创建时间，便于回滚和审计。
6. `SkillDraftAgent` 和 `RuleDraftAgent` 的输出默认是草稿，不应直接覆盖用户已有知识文件。

## 5. 推荐实施计划

本轮一次性完成 DuckDB 知识索引、共享搜索服务和主要 Agent 接入，不再先做 Python 搜索升级、后续再评估 DuckDB。

### 5.1 修正现有注入边界

1. 新增共享 helper：加载 `alwaysApply != false` rules。
2. `DataAgent`、`DataRecAgent`、`DataTransformationAgent` 统一使用该 helper。
3. `DataAgent` 自动搜索排除已注入的 always rules。
4. 为 `alwaysApply: false` 增加测试：不进入 system prompt，但相关搜索可命中。

### 5.2 新增 DuckDB 知识索引

1. 从 `rules`、`skills`、`experiences` Markdown 构建 `knowledge_index`。
2. 索引字段包含 category、path、title、description、tags、source、body_prefix、always_apply、created_at、updated_at。
3. DuckDB 查询实现多词匹配、字段权重、category scope、always rules 过滤、find/group/rank。
4. 搜索结果返回 `score`、`match_fields`、`matched_tokens`，用于调试和测试。
5. 保留安全 Python fallback，但 fallback 只用于 DuckDB 不可用时的降级，不作为主实现路线。
6. 统一 `list_all()` / `search()` 文件读取风格为 `ConfinedDir.read_text()`。
7. 修复 `test_relevant_knowledge_injected` 的弱断言。

### 5.3 抽象 `KnowledgeSearchService`

1. 把 query 构建、category scope、去重、summary block 格式化集中到一个服务。
2. 各 Agent 只声明自己需要的 categories 和 query source：
   - `DataAgent`: user question + table names + focused thread summary。
   - `DataLoadingAgent`: latest user message + attachment/file context。
   - `InteractiveExploreAgent`: start question + table names + focused thread。
   - `ChartInsightAgent`: chart type + field names + table names。
   - `ReportGenAgent`: report prompt + thread summaries。
3. 所有搜索失败都优雅降级，记录 `logger.warning(..., exc_info=True)`。

### 5.4 新增 `KnowledgeOrchestratorAgent`

1. 定义 `KnowledgeSearchPlan` / `KnowledgeSearchBundle` 数据结构。
2. 实现只读任务并行、写任务串行、全局并发上限和单任务超时。
3. 实现 `find` / `grep` / `group` / `summary` 多模式调度。
4. 实现结果去重、截断标记、分页信息和 diagnostics。
5. 预留 `KnowledgeBriefAgent` 压缩步骤，但本轮默认关闭。

### 5.5 扩展其他主流程 Agent

按优先级接入：

1. `InteractiveExploreAgent` / `ChartInsightAgent` 增加 always rules。
2. `DataLoadingAgent` 改为动态搜索 skills，并评估 always rules。
3. `ReportGenAgent` 增加 always rules + experiences。
4. `DataCleanAgentStream` 如果仍是主用入口，增加 rules + skills。
5. `CodeExplanationAgent` 只考虑 always rules。
