> **状态**: Draft  
> **创建日期**: 2026-04-24  
> **关联**: `12-unified-error-handling.md` — P5  
> **分支**: `feature/plugin-architecture`

## 1. 问题回顾

`12-unified-error-handling.md` Phase 1-4 完成后，错误响应**格式**已基本统一（`AppError` + `stream_error_event`），
但 **HTTP 状态码**仍存在三种并存模式：

| 模式 | 使用场景 | 表现 |
|------|----------|------|
| A — RESTful | `tables.py`、`data_connector.py`、`sessions.py`、`credentials.py` | `AppError.status_code` → 全局处理器 → HTTP 4xx/5xx |
| B — 全 200 + body error | `agents.py` 中 6 个非流式端点 | `jsonify({status: "error", ...})` 无第二参数 → HTTP 200 |
| C — 流式全 200 | 5 个流式端点 | `Response(stream_with_context(gen))` → HTTP 200，校验错误也在流内 |

**核心矛盾**：同一个应用内，`GET /api/tables/sample-table` 表不存在返回 404，而 `POST /api/agent/derive-data` LLM 调用失败返回 200 —— 前端无法建立一致的错误处理策略。

## 2. 决策：非流式采用 RESTful，流式实行分层错误

### 2.1 非流式端点 → 统一 RESTful 风格

**选型理由**：

1. **已完成 80%**。Phase 4 后 `tables.py`（9 处）、`data_connector.py`（7 处）、`sessions.py`、`credentials.py` 全部使用 `AppError` + 正确 HTTP 码。仅 `agents.py` 中 6 个非流式端点仍返回 200。
2. **前端已有 `!response.ok` 检查**（`apiClient.ts:175`、`useFormulateData.ts:241`、`DataLoadingChat.tsx:726` 等），全 200 方案会让这些检查失效。
3. **不需要穿越外部网关**。本项目是单体 Flask + React SPA 架构，全 200 的"网关透明"优势不适用。
4. **开发者工具可读性**。4xx/5xx 在 Network 面板中红色高亮，比全 200 + 展开看 body 调试效率高一个量级。

**规则**：

| HTTP 码 | 语义 | 使用场景 |
|---------|------|----------|
| 200 | 成功 | 正常响应 |
| 400 | 客户端错误 | 请求格式错误、参数校验失败、代码执行错误 |
| 401 | 未认证 | `identity_id` 缺失 |
| 403 | 禁止 | 签名验证失败、路径越权 |
| 404 | 未找到 | 表/资源不存在 |
| 413 | 请求体过大 | 文件/代码超限 |
| 422 | 不可处理 | LLM 返回的 JSON 解析失败 |
| 500 | 内部错误 | 未预期异常 |
| 502 | 上游错误 | LLM API 调用失败（auth、rate limit、timeout 等） |

### 2.2 流式端点 → 分层错误

**原则**：

```
请求到达
  ├── 校验层（认证、格式、参数）→ 失败直接 4xx/5xx JSON，不建流
  │
  └── 校验通过 → HTTP 200 + NDJSON 流
        ├── 正常事件 → {type: "text_delta", ...}
        ├── 运行时错误 → {type: "error", error: {...}}  （流内事件）
        └── 终止事件 → {type: "done", ...}
```

**关键变化**：将校验逻辑从生成器内部提升到路由函数体内。异常在生成器外抛出时，Flask 的全局 `error_handler` 会正常介入，返回 JSON 错误 + 对应 HTTP 状态码。只有 `Response(...)` 被 `return` 之后，HTTP 200 header 才会发送。

### 3.3 `data-loading-chat` 的额外问题

此端点的错误事件格式与 `stream_error_event` 不一致：

```python
# 当前（agents.py:1248-1251）— error 是字符串
yield json.dumps({"type": "error", "error": safe_msg}) + "\n"

# 标准（stream_error_event 输出）— error 是对象
yield json.dumps({"type": "error", "error": {"code": "...", "message": "...", "retry": false}}) + "\n"
```

应改为 `yield stream_error_event(classify_and_wrap_llm_error(e))`。

## 4. 改造方案

### 4.1 非流式端点改造模式

**Before**（以 `process-data-on-load` 为例）：

```python
@agent_bp.route('/process-data-on-load', methods=['GET', 'POST'])
def process_data_on_load_request():
    if request.is_json:
        content = request.get_json()
        token = content["token"]
        # ...
        try:
            candidates = agent.run(content["input_data"])
            response = flask.jsonify({"status": "ok", "token": token, "result": candidates})
        except Exception as e:
            logger.exception(e)
            response = flask.jsonify({                          # ← HTTP 200!
                "token": token, "status": "error",
                "result": [], "error_message": classify_llm_error(e)
            })
    else:
        response = flask.jsonify({                              # ← HTTP 200!
            "token": -1, "status": "error",
            "result": [], "error_message": "Invalid request format"
        })
    return response
```

**After**：

```python
@agent_bp.route('/process-data-on-load', methods=['GET', 'POST'])
def process_data_on_load_request():
    if not request.is_json:
        raise AppError(ErrorCode.INVALID_REQUEST, "Invalid request format", status_code=400)

    content = request.get_json()
    token = content["token"]
    # ...
    try:
        candidates = agent.run(content["input_data"])
        return flask.jsonify({"status": "ok", "token": token, "result": candidates})
    except Exception as e:
        logger.error("Error in process-data-on-load", exc_info=e)
        raise classify_and_wrap_llm_error(e)                    # ← HTTP 502!
```

> `classify_and_wrap_llm_error` 返回 `AppError(status_code=502)`，全局处理器自动
> 生成 `{status: "error", error: {code, message, retry}}` + HTTP 502。

**前端兼容性**：

- `parseApiResponse`（`apiClient.ts:76-98`）通过 `body.status === 'error'` 检测错误 → 不受 HTTP 码变化影响
- `dfSlice.tsx` 中的 thunk 直接 `response.json()` 后检查 `status` 字段 → 同样不受影响
- 唯一变化：`response.ok` 从 `true` 变为 `false`，但调用方本身没有依赖 `response.ok` 来判断这些端点

### 4.2 流式端点改造模式

**Before**（以 `data-agent-streaming` 为例）：

```python
@agent_bp.route('/data-agent-streaming', methods=['GET', 'POST'])
def data_agent_streaming():
    def generate():
        if request.is_json:                                     # ← 校验在生成器内
            content = request.get_json()
            # ...
            identity_id = get_identity_id()
            if not identity_id:
                yield stream_error_event(AppError(              # ← HTTP 已经是 200
                    ErrorCode.AUTH_REQUIRED, "Identity ID required", status_code=401,
                ), token=token)
                return
            # ... 业务逻辑 ...
        else:
            yield stream_error_event(AppError(                  # ← HTTP 已经是 200
                ErrorCode.INVALID_REQUEST, "Invalid request format", status_code=400,
            ))

    return Response(stream_with_context(_with_warnings(generate())), mimetype='application/x-ndjson')
```

**After**：

```python
@agent_bp.route('/data-agent-streaming', methods=['GET', 'POST'])
def data_agent_streaming():
    # ---- 握手前：校验层（异常走全局 error_handler → 4xx/5xx JSON）----
    if not request.is_json:
        raise AppError(ErrorCode.INVALID_REQUEST, "Invalid request format", status_code=400)

    content = request.get_json()
    token = content.get("token", "")
    identity_id = get_identity_id()

    if not identity_id:
        raise AppError(ErrorCode.AUTH_REQUIRED, "Identity ID required", status_code=401)

    # ... 其他参数提取（不会抛异常的部分也可以提前）...

    # ---- 握手后：建流（从 return Response 开始才 200）----
    def generate():
        try:
            agent = DataAgent(client=client, workspace=workspace, ...)
            for event in agent.run(...):
                yield json.dumps({"token": token, "status": "ok", "result": event}) + '\n'
        except Exception as e:
            logger.error("Error in data-agent-streaming", exc_info=e)
            yield stream_error_event(classify_and_wrap_llm_error(e), token=token)

    return Response(stream_with_context(_with_warnings(generate())), mimetype='application/x-ndjson')
```

**前端兼容性**：

`streamRequest`（`apiClient.ts:168-188`）已有 `!response.ok` 分支，会尝试解析 JSON body 中的 `error` 对象并抛出 `ApiRequestError`。这正是分层错误的前端消费路径。

直接使用 `fetchWithIdentity` 的消费者（`SimpleChartRecBox.tsx`、`DataLoadingChat.tsx`、`useFormulateData.ts`）也已有 `!response.ok` 检查：

| 消费者 | 文件:行 | 当前 `!response.ok` 处理 | 改造需求 |
|--------|---------|------------------------|----------|
| `data-agent-streaming` | `SimpleChartRecBox.tsx:406` | `throw new Error(...)` → 外层 `.catch` | ✅ 已兼容，HTTP 400/401 直接走 catch |
| `get-recommendation-questions` | `SimpleChartRecBox.tsx:1042` | 同上 | ✅ 已兼容 |
| `generate-report-chat` | `SimpleChartRecBox.tsx:1225` | 同上 | ✅ 已兼容 |
| `data-loading-chat` | `DataLoadingChat.tsx:726` | `throw new Error(...)` → 外层 `.catch` | ✅ 已兼容 |
| `get-recommendation-questions` | `useFormulateData.ts:241` | `throw new Error(...)` | ✅ 已兼容 |

**结论**：前端无需额外改造即可消费分层错误。HTTP 4xx/5xx 走现有的 `!response.ok` → catch 路径；HTTP 200 + 流内 `{type: "error"}` 走现有的 NDJSON 解析路径。

### 4.3 全局处理器已就绪

`error_handler.py` 中的 `register_error_handlers` 已覆盖：

| 处理器 | HTTP 码 | 响应体格式 |
|--------|---------|-----------|
| `AppError` | `e.status_code`（400/401/403/404/502/500 等） | `{status: "error", error: {code, message, retry}}` |
| `413 RequestEntityTooLarge` | 413 | `{status: "error", error: {code: "FILE_TOO_LARGE", ...}}` |
| `404 NotFound` | 404（`/api/*`）或 SPA 回退 | `{status: "error", error: {code: "NOT_FOUND", ...}}` |
| `Exception` 兜底 | 500 | `{status: "error", error: {code: "INTERNAL_ERROR", ...}}` |

**无需修改全局处理器**，改造仅涉及路由层。

```python
# Before — error 是字符串

# After — 使用标准 stream_error_event
yield stream_error_event(classify_and_wrap_llm_error(e))
```

**Flask 行为验证**：

需要确认 Flask 的 `stream_with_context` 在 `Response(...)` 被 `return` 之前不会提前发送 header。
标准 Flask 行为是：`return Response(...)` 后 WSGI 服务器才开始发送 200 header 和 body —— 因此路由函数体内的 `raise` 确实会被 `errorhandler` 捕获并返回 4xx/5xx。

验证方式：

```python
def test_stream_endpoint_rejects_non_json(client):
    """Non-JSON request to streaming endpoint should return 400, not 200+NDJSON error."""
    response = client.post('/api/agent/data-agent-streaming',
                           data='not json', content_type='text/plain')
    assert response.status_code == 400
    assert response.content_type == 'application/json'
    data = response.get_json()
    assert data['error']['code'] == 'INVALID_REQUEST'

def test_stream_endpoint_rejects_missing_identity(client):
    """Missing identity_id should return 401 before stream starts."""
    response = client.post('/api/agent/data-agent-streaming', json={...})
    assert response.status_code == 401
    data = response.get_json()
    assert data['error']['code'] == 'AUTH_REQUIRED'

def test_stream_endpoint_llm_error_in_stream(client, mock_llm_timeout):
    """LLM error during streaming should arrive as NDJSON error event with HTTP 200."""
    response = client.post('/api/agent/data-agent-streaming', json={...})
    assert response.status_code == 200
    lines = response.data.decode().strip().split('\n')
    last_event = json.loads(lines[-1])
    assert last_event['type'] == 'error'
    assert last_event['error']['code'] == 'LLM_TIMEOUT'
```

## 6. 风险评估

| 风险 | 影响 | 概率 | 缓解 |
|------|------|------|------|
| 非流式 HTTP 码变化导致前端行为差异 | 低 | 低 | 前端通过 `body.status` 判断，不依赖 `response.ok`；`parseApiResponse` 向后兼容 |
| 流式校验外提后 `request` 上下文丢失 | 中 | 低 | `stream_with_context` 已处理；但外提后参数通过闭包传入生成器，不再在生成器内读 `request` |
| `get_client()` 可能抛异常（model 配置无效） | 中 | 中 | 这类错误外提后会变成 HTTP 500（兜底），比之前的 200+NDJSON 错误更合理 |
| `data-loading-chat` 错误格式变化 | 低 | 中 | `DataLoadingChat.tsx:726` 的 `!response.ok` 检查已覆盖握手前错误；流内错误需确认前端解析兼容 `error` 对象格式 |
