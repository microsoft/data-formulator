## 1. 问题回顾

`12-unified-error-handling.md` Phase 1-4 完成后，错误响应**格式**已基本统一（`AppError` + `stream_error_event`），
但 **HTTP 状态码**仍存在三种并存模式：

| 模式 | 使用场景 | 表现 |
|------|----------|------|
| A — RESTful | `tables.py`（20 处）、`sessions.py`（21 处）、`data_connector.py`（9 处）、`credentials.py`（4 处）、`agents.py` 部分（17 处） | `return jsonify(...), 4xx` 或 `raise AppError(status_code=4xx)` |
| B — 全 200 + body error | `agents.py` 中 4 个原始端点 | `jsonify({status: "error", ...})` 无第二参数 → HTTP 200 |
| C — 流式全 200 | 5 个流式端点 | `Response(stream_with_context(gen))` → HTTP 200，校验错误也在流内 |

**核心矛盾**：同一个应用内，`GET /api/tables/sample-table` 表不存在返回 404，而 `POST /api/agent/derive-data` LLM 调用失败返回 200 —— 前端被迫维护两条判断路径（`!response.ok` + `body.status === 'error'`）。

## 2. 决策：统一采用 200 + body error

### 2.1 设计哲学

**HTTP 状态码描述的是传输层语义，不是应用层语义。**

HTTP 状态码的原始设计意图：

| 层级 | 谁在回答 | 含义 |
|------|---------|------|
| 传输层 | Web Server / WSGI / 网关 | "你的请求我没法处理" → 4xx/5xx |
| 应用层 | Flask 路由代码 | "你的请求我处理完了，结果在 body 里" → 200 |

**判断标准：应用代码是否有机会主动构造响应？**

- **有** → 200。服务器正常工作了，返回了结构化的回答（无论业务成功或失败）
- **没有**（服务器崩溃 / URL 无匹配路由 / WSGI 拒绝请求）→ 非 200

这与 GraphQL、gRPC-web、tRPC 的设计哲学一致：HTTP 只是传输通道，应用层成功/失败完全由 body 决定。

### 2.2 选型理由

1. **语义正确**。`POST /api/agent/derive-data` 的 LLM 超时不是 HTTP 502（Flask 不是网关），代码执行失败不是 HTTP 400（请求格式完全合法）。强行映射到 HTTP 码会扭曲语义。

2. **与原始设计一致**。微软原始作者（Chenglong Wang）在最初版本中全部使用 200 + body error。后来拆分为 routes 时，CRUD 端点引入了 RESTful 风格，但 4 个核心 agent 端点始终保持 200。统一到 200 是回归原始设计意图。

3. **网关安全**。用户会将 DF 部署在 nginx / Traefik / Cloudflare / Azure EasyAuth 等反向代理后面（SSO 架构设计已明确支持这些场景）。某些 WAF 拦截 502/503 并返回自己的错误页、nginx `error_page` 吞掉 JSON body、CDN 缓存 404 响应——这些中间层都不会干扰 200 响应。

4. **消除状态码映射争论**。当前代码中已存在语义混乱：`code-expl` LLM 格式错误 → 400，`workspace-summary` LLM 失败 → 500，`nl-to-filter` JSON 解析失败 → 422，`derive-data` 提议用 502——同类错误映射到不同 HTTP 码，每新增一种错误都要开辩论。全 200 彻底消除这个问题。

5. **前端统一判断路径**。当前前端被迫维护 `!response.ok`（HTTP 码判断）和 `body.status === 'error'`（body 判断）两条路径。全 200 后统一为单一路径：解析 body → 检查 `status` 字段。

6. **DevTools 可读性通过 `error.code` 补偿**。全 200 失去了 Network 面板红色高亮，但 body 中的 `error.code`（如 `LLM_TIMEOUT`、`TABLE_NOT_FOUND`）提供了比 HTTP 码更精确的错误信息。配合浏览器搜索功能或 console 日志，调试效率不会降低。

### 2.3 不可控的非 200（保留）

以下 HTTP 码**不受应用代码控制**，保持不变：

| HTTP 码 | 谁产生的 | 场景 |
|---------|---------|------|
| 404 | Flask 路由层 | URL 不匹配任何路由处理器（保留 SPA 回退） |
| 413 | WSGI / Flask | 请求体超过 `MAX_CONTENT_LENGTH`，在路由代码执行前被拒 |
| 500 | Flask 兜底 | 未捕获的异常逃逸到框架层（意味着我们有 bug） |

> **注意**：`error_handler.py` 中的 `@app.errorhandler(Exception)` 兜底处理器会捕获大部分未预期异常并返回结构化 JSON。只有 Flask 框架内部错误（如中间件崩溃）才会触发真正的非结构化 500。

### 2.4 流式端点 → 分层错误（不变）

流式端点的设计不受本次决策影响。分层原则保持不变：

```
请求到达
  ├── 校验层（认证、格式、参数）→ 失败返回 200 JSON（不建流）
  │   {"status": "error", "error": {"code": "INVALID_REQUEST", ...}}
  │
  └── 校验通过 → 200 + NDJSON 流
        ├── 正常事件 → {type: "text_delta", ...}
        ├── 运行时错误 → {type: "error", error: {...}}
        └── 终止事件 → {type: "done", ...}
```

校验逻辑仍需从生成器内部提升到路由函数体内（参见 §4.2）。唯一变化是：校验失败时返回 200 JSON 而非 4xx JSON。前端通过 `body.status === 'error'` 判断，不再依赖 `!response.ok`。

## 4. 改造方案

### 4.1 Step 1: 全局 `AppError` 处理器改为 200（一行改动）

**这是最高杠杆的改动**——修改一行代码，立即让所有 `raise AppError(...)` 路径统一返回 200：

**Before**：

```python
@app.errorhandler(AppError)
def _handle_app_error(e: AppError):
    body = {"status": "error", "error": e.to_dict(include_detail=app.debug)}
    return jsonify(body), e.status_code          # ← 返回 4xx/5xx
```

**After**：

```python
@app.errorhandler(AppError)
def _handle_app_error(e: AppError):
    body = {"status": "error", "error": e.to_dict(include_detail=app.debug)}
    return jsonify(body), 200                    # ← 统一 200
```

**影响范围**：`data_connector.py` 的 9 处 `raise AppError` + `tables.py` 的 2 处 `raise AppError` 立即生效。

> `AppError.status_code` 字段保留在类定义中，用于日志记录和前端 diagnostics（如果将来需要），但不再用于 HTTP 响应码。

### 4.2 Step 2: 流式端点校验外提（5 个端点）

校验逻辑从生成器内部提升到路由函数体内。但与原方案不同，校验失败时返回 **200 JSON** 而非 4xx JSON。

**Before**（以 `data-agent-streaming` 为例）：

```python
@agent_bp.route('/data-agent-streaming', methods=['GET', 'POST'])
def data_agent_streaming():
    def generate():
        if request.is_json:                                     # ← 校验在生成器内
            content = request.get_json()
            identity_id = get_identity_id()
            if not identity_id:
                yield stream_error_event(AppError(              # ← HTTP 已经是 200
                    ErrorCode.AUTH_REQUIRED, "...", status_code=401,
                ), token=token)
                return
            # ... 业务逻辑 ...
        else:
            yield stream_error_event(AppError(                  # ← HTTP 已经是 200
                ErrorCode.INVALID_REQUEST, "...", status_code=400,
            ))

    return Response(stream_with_context(_with_warnings(generate())), mimetype='application/x-ndjson')
```

**After**：

```python
@agent_bp.route('/data-agent-streaming', methods=['GET', 'POST'])
def data_agent_streaming():
    # ---- 校验层：失败返回 200 JSON（不建流）----
    if not request.is_json:
        return jsonify({"status": "error", "error": {
            "code": ErrorCode.INVALID_REQUEST,
            "message": "Invalid request format",
            "retry": False,
        }})

    content = request.get_json()
    token = content.get("token", "")
    identity_id = get_identity_id()

    if not identity_id:
        return jsonify({"status": "error", "error": {
            "code": ErrorCode.AUTH_REQUIRED,
            "message": "Identity ID required",
            "retry": False,
        }})

    # ---- 校验通过：建流（从 return Response 开始才是 NDJSON）----
    def generate():
        try:
            agent = DataAgent(client=client, workspace=workspace, ...)
            for event in agent.run(...):
                yield json.dumps({"token": token, "status": "ok", "result": event}) + '\n'
        except Exception as e:
            logger.error("Error in data-agent-streaming", exc_info=e)
            yield stream_error_event(classify_and_wrap_llm_error(e), token=token)

    return Response(stream_with_context(_with_warnings(generate())), mimetype='application/x-ndjson')
```

**关键区别**：校验失败返回的是普通 200 JSON（`Content-Type: application/json`），而非 NDJSON 流。前端通过 `response.headers.get('content-type')` 或直接解析 body 来区分。

### 4.3 Step 3: 迁移 `return jsonify(...), 4xx`（62 处）

将现有的手动返回 HTTP 码的代码逐步迁移。有两种迁移方式：

**方式 A：最小改动 — 去掉第二参数**

```python
# Before
return jsonify({"status": "error", "message": "Table name is required"}), 400

# After — 去掉 , 400
return jsonify({"status": "error", "message": "Table name is required"})
```

**方式 B：迁移到 raise AppError（推荐，享受全局处理器的统一格式）**

```python
# Before
return jsonify({"status": "error", "message": "Table name is required"}), 400

# After — raise AppError（全局处理器返回 200 + 统一格式）
raise AppError(ErrorCode.VALIDATION_ERROR, "Table name is required")
```

**优先级排序**：

| 优先级 | 文件 | 处数 | 理由 |
|--------|------|------|------|
| P1 | `agents.py` | 17 | 最易触发、用户最常遇到 |
| P2 | `tables.py` | 20 | CRUD 高频操作 |
| P3 | `sessions.py` | 21 | 生命周期操作 |
| P4 | `credentials.py` | 4 | 低频 |

### 4.4 Step 4: 前端 `!response.ok` 迁移（9 处）

**`apiClient.ts` — `streamRequest`**

```typescript
// Before
if (!response.ok) {
    let apiError: ApiError;
    try {
        const body = await response.json();
        apiError = body.error ?? { code: 'HTTP_ERROR', message: `HTTP ${response.status}`, retry: false };
    } catch {
        apiError = { code: 'HTTP_ERROR', message: `HTTP ${response.status}`, retry: false };
    }
    throw new ApiRequestError(apiError, response.status);
}

// After — 检查 body 而非 HTTP 码
// 对于流式端点，校验失败时后端返回普通 JSON（非 NDJSON）
const contentType = response.headers.get('content-type') ?? '';
if (contentType.includes('application/json')) {
    const body = await response.json();
    if (body.status === 'error') {
        const apiError: ApiError = body.error ?? {
            code: 'UNKNOWN',
            message: body.error_message ?? body.message ?? 'Unknown error',
            retry: false,
        };
        throw new ApiRequestError(apiError, response.status);
    }
}
```

**`SimpleChartRecBox.tsx` / `DataLoadingChat.tsx` / `useFormulateData.ts` — 直接 fetch 消费者**

这些地方的 `!response.ok` 检查在全 200 后永远为 `false`，变成了死代码。有两种处理方式：

1. **保守方案**：保留 `!response.ok` 作为防御性检查（处理 500 等不可控错误），同时在后面增加 body 状态检查
2. **激进方案**：直接移除 `!response.ok`，统一通过 NDJSON 解析路径处理错误

推荐保守方案——`!response.ok` 仍能捕获真正的服务器崩溃（500），不会造成误判：

```typescript
// 保留 !response.ok 作为防御层（只捕获真正的 500）
if (!response.ok) throw new Error(`HTTP ${response.status}`);
// body-level 错误由 NDJSON 解析路径中的 {type: "error"} 事件处理
```

> 由于我们的 500 兜底处理器返回结构化 JSON，`!response.ok` 只在极端情况（WSGI/nginx 层面错误）触发。保留它是零成本的防御性编程。

### 4.5 Step 5: `data-loading-chat` 错误格式统一

```python
# Before — error 是字符串
yield json.dumps({"type": "error", "error": safe_msg}) + "\n"

# After — 使用标准 stream_error_event
yield stream_error_event(classify_and_wrap_llm_error(e))
```

### 4.6 全局处理器最终状态

```python
def register_error_handlers(app: flask.Flask) -> None:

    # AppError → 200 + 结构化 body
    @app.errorhandler(AppError)
    def _handle_app_error(e: AppError):
        logger.warning("AppError [%s]: %s", e.code, e.message)
        body = {"status": "error", "error": e.to_dict(include_detail=app.debug)}
        return jsonify(body), 200

    # 413 — WSGI 层拒绝（应用代码无机会介入）
    @app.errorhandler(413)
    def _handle_413(e):
        return jsonify({
            "status": "error",
            "error": {"code": "FILE_TOO_LARGE", "message": "File too large", "retry": False},
        }), 413  # 保持 413：这是 WSGI 层行为，不是应用层

    # 404 — Flask 路由层（保留 SPA 回退）
    @app.errorhandler(404)
    def _handle_404(e):
        if request.path.startswith("/api/"):
            return jsonify({
                "status": "error",
                "error": {"code": "NOT_FOUND", "message": "Resource not found", "retry": False},
            }), 404  # 保持 404：URL 不存在是路由层事实
        return app.send_static_file("index.html")

    # 兜底 — 未捕获的异常（意味着有 bug）
    @app.errorhandler(Exception)
    def _handle_unexpected(e):
        logger.error("Unhandled exception", exc_info=e)
        error_body = {"code": "INTERNAL_ERROR", "message": "An unexpected error occurred", "retry": False}
        if app.debug:
            error_body["detail"] = traceback.format_exc()
        return jsonify({"status": "error", "error": error_body}), 500  # 保持 500：这是 bug
```

## 7. 风险评估

| 风险 | 影响 | 概率 | 缓解 |
|------|------|------|------|
| 前端 `!response.ok` 变为死代码 | 低 | 高 | 保留作为防御层，处理 500/413 等不可控错误 |
| 监控工具无法按 HTTP 码统计错误率 | 中 | 中 | 后端日志保留 `AppError.code`；可配置日志分析基于 `error.code` 统计 |
| 流式校验失败时 content-type 从 NDJSON 变为 JSON | 低 | 中 | 前端 `streamRequest` 增加 content-type 检查 |
| 62 处 `return jsonify(...), 4xx` 迁移量大 | 低 | 低 | 按优先级分步迁移，向后兼容（去掉第二参数即可） |
| `get_client()` 可能抛异常（model 配置无效） | 中 | 中 | 外提后由路由函数 try/except 捕获，返回 200 + 错误 body |

| 端点/模式 | 原因 |

## 10. 部署建议

用户将 DF 部署在反向代理后面时，建议在文档中提供以下 nginx 配置参考：

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:5567;

    # 全 200 模式下以下配置不是必须的，但作为防御性配置仍然推荐
    proxy_intercept_errors off;     # 不要让 nginx 替换后端的 JSON 错误响应
    proxy_buffering off;            # 流式端点需要关闭缓冲
}
```

> 全 200 模式的核心优势之一：即使用户**没有**配置 `proxy_intercept_errors off`，nginx 也不会拦截 200 响应。而 RESTful 模式下，如果 nginx 配置了 `error_page 502 /502.html`，就会吞掉后端精心构造的 JSON 错误。
