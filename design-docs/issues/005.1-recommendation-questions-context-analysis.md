# 15.3 推荐问题生成上下文机制深度分析

> 前置文档：[005-recommendation-questions-slow-inspect-round.md](./issues/005-recommendation-questions-slow-inspect-round.md)
> 状态：已分析，待优化
> 最后更新：2026-04-27

---

## 1. 分析背景

Issue-005 记录了 `get-recommendation-questions` 接口高达 **46–62 秒** 的延迟，主瓶颈定位在 `InteractiveExploreAgent._run_inspect_round()` 的同步多轮 LLM 调用。本文从三个维度对根因进行延展分析：

1. 上下文窗口是否被人为限制
2. 信息传递是否完整
3. 系统是否存在上下文压缩机制

---

## 2. 上下文窗口限制分析

### 2.1 没有显式的上下文窗口限制

在 `InteractiveExploreAgent` 与 `Client` 中，**不存在任何 `max_tokens`、`token_budget`、`context_window` 或 `truncate_messages` 的设置**。

- `Client.get_completion()` 仅传递 `messages`、`model`、`stream` 和 `drop_params=True`
- `_call_llm_with_tools()` 同样没有截断逻辑，完整 messages 全部送入 LLM

这意味着上下文窗口没有被人为设小，但也意味着**没有任何保护机制**来防止 prompt 膨胀。

### 2.2 隐性的"软限制"——字符截断（仅存在于 DataAgent）

系统中有两处输出侧的截断，但均不在 `InteractiveExploreAgent` 路径：

| 位置 | 截断逻辑 | 作用 |
|------|---------|------|
| `data_agent.py:1734` | tool 结果 >3000 字符截断 | DataAgent 的日志快照 |
| `data_agent.py:1740` | 普通消息 >4000 字符截断 | DataAgent 的日志快照 |
| `context.py:206` | inspect 结果 ≤3 表时 5000 字符，>3 表时 3000 字符 | `handle_inspect_source_data` 返回内容 |

**关键发现**：`InteractiveExploreAgent` 的 `_run_inspect_round()` 循环中，messages 随着每轮 tool result 的追加而**线性增长**，没有任何压缩或截断。第三轮时 messages 可能包含：

- System prompt（~1.5K 字符）
- 初始 user context（~2–5K 字符，取决于表数量）
- Round 1 assistant tool_call + tool result（~3–5K 字符）
- Round 2 assistant tool_call + tool result（~3–5K 字符）

这导致**越往后轮次，prompt 越长，LLM 处理越慢**。

### 2.3 结论

上下文窗口**没有被人为设小**，但缺乏膨胀控制机制，使得多轮 inspect 后的 prompt 体积成为隐性性能杀手。

## 3. 信息传递完整性分析

### 3.1 前端 → 后端的信息传递

在 `SimpleChartRecBox.tsx:385` 中，`getIdeasFromAgent()` 发送的请求体：

```typescript
{
    input_tables: sourceTables.map(t => ({
        name: t.virtual?.tableId || t.id.replace(/\.[^/.]+$/, ""),
        rows: t.rows,                    // ← 完整行数据
        attached_metadata: t.attachedMetadata
    })),
    exploration_thread: explorationThread,
    agent_exploration_rules: agentRules.exploration,
}
```

注意：`rows: t.rows` 传递了**完整的前端行数据**。但在后端 `run()` 中：

- `input_tables` 被传给 `build_lightweight_table_context()`
- 该函数**完全不使用 `rows` 字段**，而是直接读 parquet 文件：`workspace.read_data_as_df(table_name)`

所以 `rows` 字段**白白传输但未被使用**。

### 3.2 `build_lightweight_table_context()` 的信息缺口

对比轻量上下文和 `generate_data_summary()` 的完整摘要：

| 信息项 | 轻量上下文 | inspect 完整摘要 | 是否导致反复查询 |
|--------|-----------|-----------------|----------------|
| 表名 / 文件名 / 行数 | ✅ | ✅ | 否 |
| 列名 + 类型 | ✅ | ✅ | 否 |
| 样本行 | ✅ 3 行 | ✅ 5 行 | 轻微 |
| 数值统计(min/max/mean) | ✅ 前 8 列 | ❌ 反而没有 | 否 |
| **每列 unique 值分布** | ❌ | ✅ 7 个代表值 | **是 — 核心缺口** |
| **列描述（metadata）** | ❌ | ✅ | **是 — 核心缺口** |
| **表描述（metadata）** | ❌ | ✅ | **是 — 核心缺口** |

### 3.3 为什么 LLM 会"反复查"？

System prompt 明确告诉 LLM：

> "Lightweight schema of available datasets. Call `inspect_source_data` if you need detailed stats or sample rows."

这相当于**主动邀请** LLM 去调用工具。而轻量上下文确实缺失了三个关键信息：

1. **unique 值分布** — LLM 不知道"类别"列只有"电子/办公"两个值，还是上百个
2. **列描述** — 不知道"利润"是"毛利润"还是"净利润"
3. **表描述** — 不知道表的业务含义

**结论：信息传递不完整**（轻量上下文故意省略了 unique 值和 metadata 描述），导致 LLM 必须通过 tool call 去"发现"这些本可以零成本提供的信息。

## 4. 上下文压缩机制分析

### 4.1 全局搜索结果：几乎没有

在整个 `py-src/data_formulator` 目录搜索 `compress`、`truncate_messages`、`token_count`、`context_limit` 等关键词：

| 机制 | 存在性 | 位置 | 说明 |
|------|--------|------|------|
| 消息截断 | ⚠️ 仅 DataAgent 日志快照 | `data_agent.py:1691` | `_snapshot_dialog()` 用于可视化展示，**不影响 LLM 调用** |
| Token 计数 | ❌ 无 | — | 没有任何地方计算 token 数 |
| 滑动窗口 / 消息丢弃 | ❌ 无 | — | 没有保留最近 N 条或最近 K token 的逻辑 |
| 摘要压缩 | ❌ 无 | — | 没有将历史消息压缩为摘要的机制 |
| 工具结果截断 | ⚠️ 轻微 | `context.py:206` | inspect 结果按字符数硬截断 |

### 4.2 DataAgent vs InteractiveExploreAgent 对比

| 特性 | DataAgent | InteractiveExploreAgent |
|------|-----------|------------------------|
| 工具轮次上限 | `max_tool_rounds = 8` | `max_rounds = 3` |
| 消息截断（LLM 输入） | ❌ 无 | ❌ 无 |
| 日志快照截断 | ✅ 工具结果>3000字符、消息>4000字符 | ❌ 无快照功能 |
| 上下文压缩 | ❌ 无 | ❌ 无 |

### 4.3 一个有趣的细节：`drop_params=True`

在 `client_utils.py:180` 中，`litellm.completion` 使用了 `drop_params=True`。这意味着：

- 如果用户配置了 `max_tokens`，它会被传递给 LLM
- 但代码本身**没有设置默认值**，完全依赖外部配置或 LLM 的默认行为

## 5. 综合结论

| 怀疑项 | 验证结果 | 严重程度 |
|--------|---------|---------|
| 上下文窗口设了固定值 | **否** — 没有固定限制，但也没有保护机制 | 中等 |
| 很多信息没传过去，导致反复查 | **是** — 轻量上下文缺失 unique 值、列描述、表描述 | **高** |
| 系统有没有上下文压缩机制 | **几乎没有** — 只有日志快照的字符截断，LLM 输入侧零压缩 | 中等 |

### 5.1 核心问题链条

```
轻量上下文故意设计为"轻量"
  → 省略了 unique 值分布和 metadata 描述
  → System prompt 明确告诉 LLM"需要详细数据就调 inspect_source_data"
  → LLM 看到不完整的上下文，决定调用工具
  → 每轮只 inspect 1 张表（保守策略）
  → N 张表 → N+1 轮 LLM 调用
  → 最后一轮纯决策，100% 浪费
  → 总延迟 30–60s
```

### 5.2 与 Issue-005 优化方案的关联

Issue-005 提出的**方案 A**（将 inspect 增量信息嵌入轻量上下文）直接击中要害：

- 把 `get_field_summary()` 的 unique 值计算移到 `build_lightweight_table_context()`
- 把 `workspace.get_metadata()` 的表/列描述读出来嵌入
- 这样 LLM 一开始就拥有完整信息，**无需 inspect round**

这与"信息没传过去导致反复查"的直觉完全一致。

---

## 6. 额外发现

### 6.1 `rows` 字段的无效传输

前端在 `input_tables` 中传递了完整的 `rows`，但后端完全不使用。这造成了不必要的网络开销（虽然与 46–62 秒的总延迟相比影响很小）。

### 6.2 `max_rounds = 3` 的"空轮"浪费

无论前面调用了几次工具，第三轮总是"LLM 判断不需要更多信息 → 不调用工具 → break"，白白消耗 10–20 秒。这是算法层面的结构性浪费。

---

## 7. 建议后续行动

1. **首选**：实施 Issue-005 方案 A — 将 inspect 增量信息（unique 值、metadata 描述）嵌入 `build_lightweight_table_context()`，彻底移除 `_run_inspect_round()`
2. **辅助**：清理前端 `input_tables` 中的 `rows` 字段，避免无效数据传输
3. **长期**：考虑在 Agent 层引入轻量的 token-aware 截断或摘要机制，防止未来 prompt 膨胀
