# 18.2 - Insight 架构重构：结构化数据优先，图片可选

> 状态：方向已定，详细设计待补充  
> 创建日期：2026-04-30  
> 前置依赖：`design-docs/18-api-error-guardrails-and-chart-insight-failure.md` Phase 1 完成  
> 相关规范：`dev-guides/1-streaming-protocol.md`

## 1. 动机

当前 `ChartInsightAgent` 把 chart PNG 当成核心输入发给 vision model，存在三个缺点：

- **成本高**：图片输入通常比结构化文本摘要消耗更多 token / 计费单位，慢模型上延迟也更明显。
- **信息不完整**：图片只包含最终视觉结果，缺少原始字段类型、聚合方式、排序、过滤、采样、缺失值、分组统计和派生 lineage。
- **稳定性差**：依赖前端渲染缓存和浏览器 canvas，图表尚未渲染、渲染失败、缩放/裁剪、主题样式变化都会影响 Agent 输入。

长期方案不应默认把图片发送给大模型。图片最多作为可选辅助证据，而不是洞察的主数据源。

## 2. 设计目标

1. 洞察输入以结构化 chart spec、数据 profile 和统计摘要为主，避免默认发送高成本且信息不完整的 chart image。
2. 支持 Table Insight：Agent 只生成数据表时也能产出洞察，不依赖 PNG。
3. 普通图表洞察和表格洞察共享同一套 profile 生成基础设施。
4. 保留 vision fallback，仅用于自定义图表、地图、profile 无法表达的视觉细节。

## 3. 洞察输入结构

```text
ChartInsightInput
├── chart
│   ├── chart_type
│   ├── encoding_map
│   ├── aggregation / bin / sort / filter / scale
│   ├── display labels
│   └── chart_spec_summary
├── data_profile
│   ├── row_count
│   ├── field types / semantic types
│   ├── numeric stats: min / max / mean / median / quantiles
│   ├── categorical stats: top values / cardinality / missing count
│   ├── temporal range / granularity
│   └── outlier candidates
├── chart_series_profile
│   ├── marks or groups after aggregation
│   ├── top / bottom categories
│   ├── trend slope / correlation / distribution shape
│   ├── notable deltas / peaks / troughs
│   └── sample points used in the visual
├── lineage
│   ├── source tables
│   ├── derive code summary
│   └── filters / transforms applied
└── optional_image
    └── low-resolution chart image only when needed
```

后端或前端先生成 deterministic profile，再让 LLM 基于这些结构化事实写标题和 takeaways。LLM 不负责"看图猜数据"，只负责把已计算出的事实组织成自然语言。

## 4. 推荐实现方式

1. **前端生成 visual profile**：复用 `prepVisTable()` 和 encoding map，得到图表实际使用的聚合后数据点。
2. **共享纯函数生成统计摘要**：按字段类型生成数值、类别、时间统计，尽量不把整表传给 LLM。
3. **后端二次校验和裁剪**：限制字段数、行数和 profile 大小，避免超长 prompt。
4. **LLM 只接收 profile + spec summary**：默认不传图片。
5. **可选 vision fallback**：只有在自定义图表、地图、复杂视觉编码或 profile 不足时，才传低分辨率图片，并记录 `image_used=true`。

## 5. 洞察 Agent 拆分

建议把洞察能力拆成两层：

1. `InsightProfiler`：确定性代码，不调用 LLM，负责从 chart/table/lineage 生成结构化事实。
2. `InsightNarratorAgent`：调用 LLM，只基于 `InsightProfiler` 的事实生成用户可读标题和 takeaways。

进一步按输入类型区分：

| Agent / Helper | 输入 | 是否需要图片 | 适用场景 |
|----------------|------|--------------|----------|
| `TableInsightProfiler` | table schema + stats + sample | 否 | Agent 只生成数据表 |
| `ChartInsightProfiler` | chart spec + visual aggregate + table stats | 否 | 普通图表洞察 |
| `InsightNarratorAgent` | profiler facts | 否 | 生成自然语言洞察 |
| `VisionInsightFallback` | profiler facts + low-res image | 可选 | 自定义图表、地图、profile 无法表达的视觉细节 |

长期目标是把图片路径降级为 fallback，而不是主路径。

### 5.1 Python 模块落点

`InsightProfiler` 不属于 `agents/`。它不是 LLM Agent，而是确定性的 profile 生成层：不持有 `client`，不调用 `get_completion()`，只把输入数据整理成可测试、可裁剪、可复用的结构化事实。

新增独立包：

```text
py-src/data_formulator/insights/
├── __init__.py
├── profile_types.py
├── table_profiler.py
├── chart_profiler.py
└── vision_fallback.py
```

模块职责：

| 模块 | 主要类 / 函数 | 职责 |
|------|---------------|------|
| `insights/profile_types.py` | `InsightProfile`、`TableProfile`、`ChartProfile`、`InsightDiagnostics` | 定义 profiler 输出的数据结构、裁剪后的统计摘要和 diagnostics 字段 |
| `insights/table_profiler.py` | `TableInsightProfiler` | 从 table rows、schema、semantic metadata 和 sample 生成表格 profile |
| `insights/chart_profiler.py` | `ChartInsightProfiler` | 从 chart type、encoding map、chart config、visual aggregate、table profile 和 lineage 生成图表 profile |
| `insights/vision_fallback.py` | `needs_vision_fallback()`、`build_vision_context()` | 判断是否需要低分辨率图片，并准备 fallback 输入；不直接调用模型 |

`agents/` 只保留真正调用模型的组件：

```text
py-src/data_formulator/agents/
├── agent_chart_insight.py        # 兼容旧入口，逐步变薄
└── agent_insight_narrator.py     # 新增，负责 LLM 文案生成
```

### 5.2 与现有代码的对接路径

当前后端入口是 `routes/agents.py` 的 `/chart-insight`，直接创建 `ChartInsightAgent`：

```text
routes/agents.py request_chart_insight()
└── ChartInsightAgent.run(chart_image, chart_type, field_names, input_tables)
```

Phase 2 不新建一个平行的 `/insight` 路由来绕开旧路径。先保留 `/chart-insight` API 作为兼容入口，把内部实现改成 structured-first：

```text
routes/agents.py request_chart_insight()
└── ChartInsightAgent.run(...)
    ├── ChartInsightProfiler / TableInsightProfiler
    ├── InsightNarratorAgent
    └── VisionInsightFallback（仅 profile 不足或图表类型必须看图时）
```

迁移原则：

1. `ChartInsightAgent` 在第一步可作为兼容门面，负责把旧参数适配为新的 profile 输入；后续稳定后再考虑重命名或拆路由。
2. `InsightNarratorAgent` 放在 `agents/`，因为它持有 LLM `client` 并负责 prompt、language instruction、knowledge injection 和 model routing。
3. `TableInsightProfiler` / `ChartInsightProfiler` 放在 `insights/`，因为它们是纯业务计算，必须能用单元测试覆盖，不依赖 Flask request、workspace identity 或 LLM client。
4. 前端负责传递 chart spec、encoding map、render metadata 和必要的 table context；后端负责统一生成和裁剪 profile，再调用 narrator。
5. 图片只在 `needs_vision_fallback()` 返回 true 时进入 prompt，并在响应 diagnostics / 日志中标记 `image_used=true`。

### 5.3 `InsightProfiler` 命名说明

这里的 `Profiler` 含义是“数据画像生成器”，不是性能 profiler。

它输出的是 LLM 可消费的事实画像，例如：

- 表级信息：行数、字段列表、字段类型、缺失值、样本行。
- 数值字段：min / max / mean / median / quantiles、异常值候选。
- 类别字段：top values、cardinality、长尾分布。
- 时间字段：范围、粒度、趋势候选。
- 图表级信息：使用的 encoding、聚合后 series、top / bottom categories、峰值、趋势、相关性。
- lineage：来源表、派生步骤、过滤和 transform 摘要。

LLM 的职责是基于这些事实生成自然语言，不再负责从图片中猜测数据。

## 6. 产品边界：Chart Insight vs Table Insight

建议明确拆成两类产品能力，并共享同一套 profile：

1. **Chart Insight**：输入 chart spec + visual aggregate profile + table stats，默认不传图片。
2. **Table Insight**：输入 table schema + sample/statistics，不依赖 PNG，适用于纯数据表。

如果产品希望"Agent 生成数据表后也自动洞察"，应新增 table insight 路径，而不是让 chart insight 在没有 chart image 时失败。对普通图表，也应优先用结构化 profile；只有 profile 无法表达视觉细节时才启用 image fallback。

## 7. 实施内容

- 新增 `py-src/data_formulator/insights/` 包，承载所有不调用 LLM 的 profile 生成逻辑。
- 新增 `TableInsightProfiler`，从 table rows、metadata、schema 和 sample 生成表格 profile，支持 Agent 只生成数据表时直接产出表格洞察。
- 新增 `ChartInsightProfiler`，从 chart spec、encoding map、visual aggregate、table profile 和 lineage 生成图表 profile。
- 新增 `InsightNarratorAgent`，接收 profiler facts 并调用 LLM 生成标题和 takeaways。
- `ChartInsightAgent` 先保留为 `/chart-insight` 的兼容门面，默认走 profile + narrator，不再默认接收 chart image。
- 自动洞察在 profile 可生成时不等待图片缓存；仅 vision fallback 场景使用 `waitForChartImage()`。
- 区分 `profile_generation_failed`、`chart_image_not_ready` 和后端失败。
- 对 `Table` 图表走 table insight profile；对 `Auto` / `?` 图表等待 chart 类型确定后再触发。
- 对自定义图表和 profile 不足的场景，保留低分辨率 image fallback，并在响应 diagnostics 中标记 `image_used`。
- 测试覆盖：profile 大小上限、字段统计正确性、聚合后数据点摘要、无图片路径的洞察生成。

## 8. 待详细设计的问题

以下问题需要在详细设计阶段解决：

1. **统计计算性能**：percentiles、correlation、trend slope 在大数据集（100K+ 行）上如何高效计算？
2. **visual aggregate 来源**：前端应尽量传递图表渲染已经使用的聚合后数据或 render metadata，避免后端复刻完整 Vega-Lite/ECharts 渲染逻辑；后端负责基于这些输入统一生成、校验和裁剪 profile。
3. **Token 预算**：profile 发送给 LLM 的 token 上限是多少？如何裁剪？
4. **Vision fallback 判定**：哪些图表类型必须使用图片？判定逻辑放在前端还是后端？
5. **chart_series_profile 生成**：聚合后的数据点只存在于渲染管线中，如何在不渲染图表的情况下获取？

## 9. Open Questions

1. 哪些图表类型必须保留 vision fallback，例如地图、自定义 mark、复杂组合图？
2. Table Insight 是否需要独立的 UI 入口，还是复用 Chart Insight 面板？

## 10. 已确定的设计决策

| 决策项 | 结论 |
|--------|------|
| `InsightProfiler` 模块位置 | 放在新包 `py-src/data_formulator/insights/`，不放在 `agents/` |
| `InsightProfiler` 含义 | 数据画像/结构化事实生成器，不是性能 profiler，也不是 LLM Agent |
| LLM 调用位置 | 只放在 `agents/agent_insight_narrator.py` 等 Agent 模块中 |
| 现有 API 对接 | 保留 `/chart-insight` 和 `ChartInsightAgent` 作为兼容入口，内部改为 profile-first |
| 前后端分工 | 前端传 chart spec / encoding / render metadata，后端生成并裁剪 profile，然后调用 narrator |
| 图片路径 | 默认不使用图片；仅 `VisionInsightFallback` 判定需要时使用低分辨率图片 |
