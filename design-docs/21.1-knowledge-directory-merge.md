## 1. 动机

`skills` 和 `experiences` 在存储格式、body 长度限制、搜索匹配逻辑、目录深度上完全一致，区别仅在于"方法 vs 案例"的语义标签。实际使用中用户很难区分。既然搜索已经统一，检索入口已经合并，不如直接合并物理目录，减少概念数量。

当前代码已测试通过但尚未投入生产，改造成本最低。

## 2. 目标结构

### 改造前

```
user_home/knowledge/
├── rules/            ← 手动创建，flat 文件，alwaysApply 语义
│   └── date-format.md
├── skills/           ← 一层子目录 + 文件
│   └── cleaning/
│       └── handle-missing.md
└── experiences/      ← 一层子目录 + 文件
    └── sales-analysis/
        └── roi-by-region.md
```

### 改造后

```
user_home/knowledge/
├── rules/            ← 不变
│   └── date-format.md
└── library/          ← 合并后的知识库（skills + experiences）
    ├── cleaning/
    │   └── handle-missing.md
    ├── sales-analysis/
    │   └── roi-by-region.md
    └── tips/
        └── common-pitfalls.md
```

目录名选择 `library`，语义清晰（"知识库"），不与外层 `knowledge/` 重名。

## 3. front matter 扩展

### 3.1 不引入 `kind` 字段

合并后不再区分 skill / experience / tip，统一视为"可复用的通用方法论"。`ExperienceDistillAgent` 提炼时应侧重通用性——提取可迁移的方法和注意事项，而非记录具体案例的细节。front matter 结构：

```yaml
---
title: 处理缺失值
tags: [cleaning, missing-values]
source: manual       # manual | distill | auto
created: 2026-05-03
updated: 2026-05-03
---
```

`source` 字段已有，足以区分来源：
- `manual`：用户手写
- `distill`：`ExperienceDistillAgent` 自动提炼
- `auto`：未来自动记忆系统生成

不需要额外分类——合并目录的初衷就是消除用户的分类负担。如果再加 `kind` 分三种，和之前三目录本质一样。

### 3.2 rules 不变

`rules/` 的 front matter 结构不变，继续使用 `description` + `alwaysApply`。

## 4. 后端改动清单

### 4.1 `knowledge/store.py` — 核心改动

**常量**

| 改造前 | 改造后 |
|--------|--------|
| `VALID_CATEGORIES = frozenset({"rules", "skills", "experiences"})` | `VALID_CATEGORIES = frozenset({"rules", "library"})` |
| `_MAX_DEPTH = {"rules": 1, "skills": 2, "experiences": 2}` | `_MAX_DEPTH = {"rules": 1, "library": 2}` |
| `KNOWLEDGE_LIMITS = {"rule_description_max": 100, "rules": 350, "skills": 2000, "experiences": 2000}` | `KNOWLEDGE_LIMITS = {"rule_description_max": 100, "rules": 350, "library": 2000}` |

**`__init__`**

```python
# 改造前
self._jails = {
    "rules": ConfinedDir(self._root.root / "rules", mkdir=True),
    "skills": ConfinedDir(self._root.root / "skills", mkdir=True),
    "experiences": ConfinedDir(self._root.root / "experiences", mkdir=True),
}

# 改造后
self._jails = {
    "rules": ConfinedDir(self._root.root / "rules", mkdir=True),
    "library": ConfinedDir(self._root.root / "library", mkdir=True),
}
```

**`_ensure_front_matter`**

非 rules 时无需额外字段，和原 skills/experiences 行为一致。

**`list_all`**

对 `library` 条目不返回额外字段。只有 `rules` 继续返回 `description` 和 `alwaysApply`。

**`KnowledgeItemMeta`**

无需新增字段。

**`search`**

无特殊改动——`cats = categories or list(VALID_CATEGORIES)` 自动适配。

### 4.2 `routes/knowledge.py` — API 路由

**distill-experience 端点**

```python
# 改造前
store.write("experiences", rel_path, md_content)
return json_ok({"category": "experiences", ...})

# 改造后
store.write("library", rel_path, md_content)
return json_ok({"category": "library", ...})
```

`rel_path` 的子目录前缀继续由 `category_hint` 决定（如 `"sales-analysis/roi.md"`），不需要改路径生成逻辑。

**其他端点（list/read/write/delete/search）**

无结构性改动——它们通过 `VALID_CATEGORIES` 校验，常量改了它们就自动适配。

### 4.3 Agent 文件

| 文件 | 改造前 | 改造后 |
|------|--------|--------|
| `data_agent.py` 工具 schema | `enum: ["rules", "skills", "experiences"]` | `enum: ["rules", "library"]` |
| `data_agent.py` `_search_relevant_knowledge` | `categories=["rules", "skills", "experiences"]` | `categories=["rules", "library"]`（或 `None` 全量） |
| `data_agent.py` `_load_knowledge_rules` | `list_all("rules")` + `read("rules", ...)` | 不变 |
| `agent_data_loading_chat.py` | `categories=["skills"]` | `categories=["library"]` |
| `agent_interactive_explore.py` | `categories=["experiences"]` | `categories=["library"]` |
| `agent_chart_insight.py` | `categories=["experiences"]` | `categories=["library"]` |
| `agent_experience_distill.py` | prompt 引导写具体案例 | 改——引导提炼通用方法论（见 §4.6） |

注意：rules 注入分两层：

- **`alwaysApply` rules → system prompt**：各 Agent 通过 `_load_knowledge_rules()` 加载，不走搜索。这条路径不变。
- **非 `alwaysApply` rules → 搜索命中**：仅 `DataAgent` 搜索时包含 rules（`categories=["rules", "library"]`），因为它有完整工具链可判断相关性。其余 Agent 只搜 `["library"]`，避免简单 Agent 误用低相关性 rules。

**避免重复注入**：`KnowledgeStore.search()` 在搜索 `rules` 类别时，应**跳过 `alwaysApply=true` 的条目**——这些条目已经通过 `_load_knowledge_rules()` 注入到 system prompt，搜索再命中会导致同一条 rule 出现两次。实现方式：在 `search()` 的遍历循环中，对 `rules` 类别的条目检查 `alwaysApply`，为 `true` 则跳过。

### 4.4 `_search_relevant_knowledge` 注入文本

```python
# 改造前
f"[{hit['category']}] {hit['title']}: {hit['snippet']}"
# 例如：[skills] Handle Missing Values: ...
# 例如：[experiences] ROI Analysis: ...

# 改造后：按 category 映射为语义标签
label = "rule" if hit["category"] == "rules" else "knowledge"
f"[{label}] {hit['title']}: {hit['snippet']}"
# 例如：[knowledge] Handle Missing Values: ...
# 例如：[rule] Date Format Convention: ...
```

使用 `[knowledge]` / `[rule]` 而非目录名 `[library]`：LLM 不需要知道目录结构，语义标签更有助于它区分"约束性规则"和"参考性知识"。

### 4.5 `routes/knowledge.py` — docstring 更新

`distill_experience()` 的 docstring 中 `sub-directory under experiences/` 改为 `sub-directory under library/`。

### 4.6 `agent_experience_distill.py` — prompt 改造

当前 `SYSTEM_PROMPT` 和 `LOG_SYSTEM_PROMPT` 的结构引导 LLM 记录具体案例（"什么数据、做了什么、发现了什么"），产出的内容复用价值有限。改造目标：让 prompt 引导 LLM **提炼通用方法论**——可迁移的方法、注意事项和踩坑规避，而非一次性的分析报告。

**改造要点**

| 项 | 改造前 | 改造后 |
|----|--------|--------|
| 角色定位 | `experience summarizer` | `knowledge distiller`——强调"蒸馏可复用知识" |
| 标题模板 | `[数据源]: [分析技术] — [关键发现]` 要求写具体数据和结果 | `[方法/技巧]: [适用场景] — [核心要点]` 强调方法的可迁移性 |
| 文档结构 | Scenario / Method / Key Findings（案例导向） | When to Use / Method / Pitfalls & Tips（方法论导向） |
| Rules 侧重 | "Explain the final successful analysis path" | "Extract the generalizable method; omit case-specific data values, column names, and row counts unless they illustrate a universal pattern" |
| source 字段 | `agent_summarized` | `distill`（与 front matter 的 source 枚举对齐） |

**改造后 prompt 结构示意**（`SYSTEM_PROMPT` 和 `LOG_SYSTEM_PROMPT` 同步改）

```
You are a knowledge distiller. Given the context of a data analysis session,
extract reusable methodology that can help with similar future tasks.

The knowledge entry must include:

1. **Title**: "[Method/technique]: [applicable scenario] — [core takeaway]".
   Focus on the method, not the specific dataset.
2. **When to Use**: describe the general conditions where this method applies.
3. **Method**: the concrete steps, abstracted from the specific dataset.
   Use generic placeholders (e.g. "the target column", "the grouping key")
   instead of actual column names when the names are not universally meaningful.
4. **Pitfalls & Tips**: gotchas encountered during the session, workarounds,
   and things to watch out for. This is the most valuable section.
5. **Tags**: keywords for future search.

Rules:
- Focus on *transferable* methods and caveats, not case-specific details.
- If a repair/retry was needed, explain *why* it failed and the general fix,
  not just what happened in this specific run.
- Keep under 500 words. Prefer concise, actionable guidance.
- ...（其余 rules 不变：安全、语言指令等）
```

这样产出的知识条目示例对比：

**改造前**（案例记录）：
> "sales_data 表：用 groupby+agg 计算各区域 ROI — 发现 Q3 最高"

**改造后**（通用方法论）：
> "分组聚合计算比率指标：当需要按维度对比一个比率型指标时 — 注意分母为零的处理"

## 8. 对 15.3 Agent 注入矩阵的影响

`15.3` 文档中按三类做的注入矩阵需要更新：

| 改造前 | 改造后 |
|--------|--------|
| 应注入 skills ✅ | 应注入 library ✅ |
| 应注入 experiences ✅ | 应注入 library ✅ |
| 应注入 skills ❌ + experiences ✅ | 应注入 library ✅（搜索结果自动包含两类） |
| 应注入 skills ✅ + experiences ❌ | 应注入 library ✅（同上） |

简化后的矩阵只有两列：rules 和 library，判断更简单。

合并后所有 Agent 搜索同一个 `library` 池，不再按 skills/experiences 分流。当前知识库规模小，搜索精度影响可忽略。未来如果需要精确控制（某 Agent 只要某类知识），通过搜索 API 增加 `tags` 过滤参数即可，不影响目录结构。

## 11. 决策记录

| 编号 | 决策 | 理由 |
|------|------|------|
| D1 | 合并目录名用 `library` | 语义清晰，不与外层 `knowledge/` 重名 |
| D2 | 不引入 `kind` 分类字段 | 合并目录就是为了消除分类负担；再加 kind 分三种和之前三目录本质一样 |
| D3 | 用已有的 `source` 字段区分来源 | `manual` / `distill` / `auto` 足以区分手写和自动生成 |
| D4 | 不写迁移代码 | 尚未投产，无存量数据；减少复杂度 |
| D5 | rules 注入分两层，library 搜索统一不分类 | `alwaysApply` rules 通过 system prompt 注入（不走搜索）；非 always rules 仅 `DataAgent` 的搜索工具可按需命中；其余 Agent 只搜 library |
| D6 | rules 保持独立目录 | rules 有 alwaysApply 强语义、flat 目录、短 body 限制，与 library 差异大 |
| D7 | 搜索排除 alwaysApply rules | 避免与 `_load_knowledge_rules()` 的 system prompt 注入重复；搜索只返回非 alwaysApply 的 rules |
| D8 | 注入标签用 `[knowledge]` / `[rule]` 而非目录名 | LLM 不需要知道目录结构；语义标签帮助区分约束性规则和参考性知识 |
| D9 | ExperienceDistillAgent prompt 改为方法论导向 | 案例记录复用价值低；通用方法论（方法+踩坑规避）才是知识库的核心资产 |
